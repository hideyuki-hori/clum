# clum

曖昧性を排除するための言語。ビルドすると Web 標準の HTML / CSS / JS が出る（現在は HTML のみ）。
**仕様が正、実装は従。** 実装の都合で仕様を曲げない。

## 仕様書（必読・すべての判断の根拠）

- @docs/spec/clum-design-v3.md — 設計原則・5層構造・ターゲットモデル
- @docs/spec/clum-core-spec.md — コア仕様（字句・型・スコープ・実行モデル）
- @docs/spec/clum-html-spec.md — HTML ターゲット仕様
- @docs/spec/clum-step1-html.md — **現在のステップのスコープ定義（最優先で読む）**
- docs/spec/latest.md — 現在形だけのチートシート（経緯なし・派生文書。正は上の各仕様書。仕様を変えたら同じコミットでこの文書も更新する）

## 仕様変更の手続き（最重要ルール）

- 実装と仕様が食い違ったら、**必ず仕様側が正**。実装を直す。
- 仕様に書かれていない挙動に出会ったら、**勝手に決めて実装しない**。
  「仕様の穴」として該当文書の要決定リストに追記し、選択肢を提示して**停止**する。
- 仕様書の「決定済み」項目を実装の都合で変更しない。変更提案は人間の判断を仰ぐ。

## ステップ1のスコープ境界（実装しないもの）

以下は仕様に存在するが、**現在のステップでは実装しない**。気を利かせて先取りしない。

- `?` マッチ・`:is`・`:else`・直和型
- `Eff` の合成（`~` / `<-`）。`Eff`/`!` は `build: Recipe -> Eff<Void>` の1箇所のみ
- `:` 呼び出し（述語・制約・中置記法）・四則演算・ジェネリクス制約境界
- js / css / wasm ターゲット、`@js` / `@rust`、`:share`、`Stream<T>`

## コマンド

```bash
cargo build
cargo test                  # golden test を含む全テスト
cargo clippy -- -D warnings # 警告ゼロを維持
cargo fmt
cargo run -- build ./examples/main.clum
```

## 実装規約

- パーサは**手書きの再帰下降**。字句解析器がインデントを INDENT/DEDENT トークンへ変換する。
  パーサジェネレータ・正規表現ベースの構文解析は使わない。
- AST の**全ノードに span（ソース位置）を必ず持たせる**。span のないノードを追加しない。
- 型検査はステップ1では単相＋`map` の intrinsic 特別扱い。HM の本実装は後続ステップ。
- エラーメッセージの品質はこの言語の価値の中心。span を使い、位置と理由を必ず示す。
- 依存クレートの追加は**事前に相談**する（理由と代替案を添えて提案し、承認を待つ）。
- CLI は `clum build <path>` の形式。エントリファイルは引数で明示し、暗黙のカレント探索をしない。

## テスト

- golden test: `tests/golden/` に入力 `.clum` と期待 `.html` の対を置く。
- 仕様の確定済み規則（エスケープ・void 要素・無挿入テキスト連結・無整形出力）は
  必ず golden test でケース化する。
- 新しい構文を実装したら、正常系と**コンパイルエラーになるべき系**（暗黙捕獲・
  シャドーイング・未知の Tag 等）の両方をテストに追加する。

## 文書規約

- Markdown のテーブル・罫線による表は**禁止**（横に長いと読みにくい）。
- 表にしたい内容はネストした箇条書きで書く:
  - 項目ごとにトップレベルの `-` を置く
  - 属性は1レベル深い `-` に「名前: 値」で書く（2列相当なら `- 項目 — 説明` の1行でよい）

## やってはいけないこと

- 仕様にない糖衣構文・省略記法・「便利機能」の追加
- 出力 HTML への整形（改行・インデント）の付与
- エラーを握りつぶしてビルドを続行すること
- テストなしで挙動を変えること
- 文書での Markdown テーブルの使用（文書規約参照）
