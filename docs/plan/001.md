### 0-1. 要素記法の正: `h .tag` か裸の要素名か

- step1 §1・§4 は `h .div`（`h` 関数 + `Tag` 直和型）を採る。
- html-spec §1・§2・§3 は裸の要素名（`div ->`）と行頭トークンによる行解釈規則を定義している。
- 選択肢: (a) ステップ1は `h .tag` のみを実装し、html-spec の裸要素名記法は改訂・後続ステップ扱いとする / (b) 裸要素名を正とし step1 の例を修正する。
- **決定: (a)。`h .tag` 方式を正とし、裸要素名の記法は廃止（html-spec を改訂）。**

### 0-2. `h` 適用と子ブロックの構文（`->` の要否）

- step1 §1 の例は `h .html` の直後に `->` なしでインデントブロック（子）が続く。
- html-spec §3.1 は「`名前 属性... ->` の後のインデントブロックが子」と定義しており食い違う。
- コア構文には「関数適用の直下のインデントブロックを引数として渡す」規則が存在しないため、どちらを採るにせよ html ターゲットの構文規則としての明文化が必要。
- 選択肢: (a) step1 の例を正とし「`h` 適用行の直下のインデントブロックは子リスト」と仕様化（`->` は書かない）/ (b) html-spec に合わせ `->` を必須とし step1 の例を修正する。
- **決定: (a)。`->` なし。レコード構築（コア §2.7）と同じ「行＋1レベル深いインデントブロック」の形として html 仕様 §3.1 に明文化。**

### 0-3. 属性の構文と `h` の型

- `h .meta charset 'utf-8'` の `charset 'utf-8'` は「属性名 値」のペアだが、コアの関数適用（可変長引数なし）では型付けできない。
- 選択肢: (a) `h` を関数値ではなく html ターゲットが与える**特別形式**（属性ペア列と子ブロックを取る専用構文）として仕様化する / (b) 属性を `Vec<Attr>` 等の通常の値として脱糖する規則を定義する。
- (a) の場合、`h` を部分適用・値渡しできるか（できないなら明記）も併せて決定する。
- **決定: (a)。`h` は html ターゲットの特別形式とし、部分適用・値渡しは不可と明記（html 仕様 §3.1）。**

### 0-4. コンポーネント呼び出しの表記

- html-spec §2.2 は「タグ位置の UpperCamelCase 識別子＝定義名参照」（行解釈規則前提）だが、0-1 で `h .tag` 方式を採る場合にこの規則をどう持ち込むかが未定義。
- 選択肢: (a) `Card 'お知らせ'` のような定義名参照タグを `h` と同じ子ブロック規則で許す / (b) ステップ1は通常の関数適用（`card 引数...`）のみとし、定義名参照タグは後続ステップへ先送り。
- **決定: (a)。定義名タグを維持し、`h` と同じ子ブロック規則を適用（html 仕様 §2.2）。子ブロックはシグネチャ末尾の `Element`（現 `Html`。または `Vec<Html>`）引数に渡される。**

### 0-5. フィールドアクセス `page.title` の字句規則

- `page.title` というフィールドアクセスは例に登場するが、構文規則として仕様書に明記されていない。`h .html`（Tag リテラル）との区別も未定義。
- 提案: 空白規則で区別する。`識別子.識別子`（密着）＝フィールドアクセス、空白を挟んだ `.識別子` ＝ Tag 等のドット前置リテラル。コア仕様への追記が必要。
- **決定: 提案どおり空白規則で区別（コア仕様 §1.7 として追記）。**

### 0-6. `Recipe.files` の型と `index = ->` の不整合

- core-spec §2.7 は `# Recipe root: String, files: Vec<Element>` と宣言するが、step1 §1 の `index = ->` はゼロ引数関数であり `Element` ではない。型が合わない。
- 選択肢: (a) `files` を「ゼロ引数関数の Vec」とし build が呼び出す / (b) `index = h .html ...` の値束縛に例を改め `files: Vec<Element>` を維持する / (c) ゼロ引数関数の適用構文を定義して `files: index!, ...` 等で渡す。
- (a)(c) はゼロ引数関数の適用構文（現仕様に存在しない）の決定を伴う。
- **決定: (b)。`files: Vec<Element>` を維持し、ページは `index = h .html ...` の値束縛にする（step1 §1 の例を修正済み）。帰結として「構築を行の途中から始めた場合、本体はその行より1レベル深いインデントで続く」規則をコア §2.7 にも追記。** → その後、第2次改訂で `index: Html = ...`（型注釈つき束縛）・`documents` 方式に発展（下記追補参照）。

### 0-7. 暗黙捕獲禁止の正確な範囲

- core-spec §6.1 はクロージャによる外側スコープの暗黙参照を禁止するが、step1 §1 の例ではラムダ内から `h` や `pages`（ファイルトップレベルの束縛・import）を直接参照している。
- 提案: 「トップレベル定義・import・prelude は常時参照可。関数本体の内側で生まれたローカル束縛と引数のみ、より内側のラムダから暗黙捕獲できない」と確定する。この線引きなしにはスコープ検査を実装できない。
- **決定: 提案どおり（コア仕様 §6.1 に追記）。**

### 0-8. `:out .html` と `@ prelude/html` の関係

- html-spec §1.1 の main.clum は `@ prelude/html` で `h, Recipe, build` を import するが、step1 §1 のページファイルは import なしで `:out .html` 内の `h` を使っている。
- 選択肢: (a) `:out .html` ブロック内では html 語彙（`h`・`Tag`・`Element`・属性）が自動で可視になる / (b) ページファイルにも `@ prelude/html` の明示 import を必須とし、`:out` は語彙の有効化（検査の切り替え）のみを担う。
- **決定: (a)。`:out` 自体が明示宣言であり、コア仕様 §9 の定義どおりの動作（html 仕様 §1 に明記）。** → その後、第2次改訂で `:out` 自体を廃止（語彙は prelude で常時可視。下記追補参照）。

### 0-9. `root: './dist'` の基準ディレクトリ

- 相対パスの基準が main.clum の場所か、コマンド実行時のカレントディレクトリかが未定義。
- 提案: main.clum のあるディレクトリを基準とする（実行場所に依存しない）。
- **決定: 提案どおり main.clum のディレクトリ基準（html 仕様 §1.1 に明記）。**

### 0-10. 出力ファイル名導出の明文化

- 「`files` に渡した定義名から導出」（html-spec §1.1）は、値になった時点で名前情報が消えるため、通常の評価では実現できない。
- 提案: 「`Recipe` 構築の `files:` の各要素は**識別子参照でなければならない**（式は不可）」という構文制約として明文化し、その識別子名を出力名にする。
- **決定: 提案どおり識別子参照のみ許可。違反はコンパイルエラー（html 仕様 §1.1 に明記）。** → その後、第2次改訂で廃止（`Document.path` に出力先を明示する方式へ置換。下記追補参照）。

## パイプライン構成

```
ソース (.clum)
  → 字句解析   … インデント→INDENT/DEDENT、全トークンに span
  → 構文解析   … 手書き再帰下降、全 AST ノードに span
  → 名前解決   … スコープ規則・シャドーイング禁止・暗黙捕獲禁止・
                 1定義=1実装・1ファイル=1export・@ import の解決
  → 型検査     … 単相 + map intrinsic、暫定 prelude の型環境、
                 Tag/属性の検査
  → 評価       … main.clum をビルド時実行。純粋式の評価と
                 `!` による効果（build = ファイル書き出し）の分離
  → HTML 出力  … エスケープ・void 要素・無挿入連結・無整形・DOCTYPE
  → dist/*.html
```

## フェーズ分割

各フェーズの完了条件は「単体テストが緑・`cargo clippy -- -D warnings` が通る」を含む。

### フェーズ1: 基盤

- CLI スケルトン（`clum build <path>`。パス引数必須、暗黙のカレント探索なし。終了コード: 成功 0 / エラー 1）
- `span.rs` / `source.rs` / `diag.rs`。エラー表示形式を確立する（位置と理由を必ず示す。例:）

```
error: `Tag` に `.dvi` はありません
 --> examples/index.clum:5:7
  |
5 |     h .dvi
  |       ^^^^ もしかして `.div` ですか？
```

- golden test ランナー（ディレクトリ走査型）をこの時点で整備する。ケースは後続フェーズで随時追加。
- **フェーズ1 完了（2026-07-04）**。既知の制限: 診断のキャレット位置は文字（コードポイント）単位で計算するため、全角文字を含む行では表示上ずれる。厳密な位置合わせには East Asian Width 表が必要（std のみ制約下では自前実装になるため、必要性が確認できてから対応を検討）。

### フェーズ2: 字句解析器

- インデント: スペースのみ・1レベル2スペース固定。タブ＝エラー、奇数／飛びインデント＝エラー。INDENT/DEDENT/NEWLINE トークン化。
- コメント `//`（行末まで。出力に残らない）。
- 文字列 `'...'`: 補間 `{式}` を字句レベルで分解、`{{` `}}` のエスケープ。`"` 文字列は存在しない。
- 数値（整数→i32 既定、小数→f64 既定。負号はリテラルの一部）。
- 識別子（lower-kebab-case / UpperCamelCase）、ドット前置リテラル（`.html` 等。0-5 の決定に従う）、コロン語（`:pub` のみ。他はエラー）。
- 記号: `#` `@` `!` `|>` `->` `=` `,` `:` `.` `{` `}`（`<` `>` は型構文 `Vec<T>` で使用）。
- リスト要素マーカー: 行頭（インデント直後）の `- `（追補の決定）。kebab-case・負数リテラルの `-` とはインデント直後＋直後の空白で区別する。
- 全トークンに span。字句単体テストを同フェーズで書く。
- **フェーズ2 完了（2026-07-05）**。追補2の4決定（Path トークン・タブ全面禁止・`}}` 対称・裸の式行廃止）を反映済み。テキスト行はパーサ向けの生取得 API（行頭分類＋行の生テキスト取得）で扱う。

### フェーズ3: AST とパーサ

- AST 全ノードに span（span のないノードを追加しない）。
- 再帰下降で以下を構文解析:
  - `#` 宣言（積型 / 関数シグネチャ。型式は 名前・`Vec<名前>`・関数型の単純形のみ）
  - 束縛 `name = 式`・型注釈つき束縛 `name: Type = 式` / 実装束縛 `name: DefName 引数... -> 本体`
  - 無名関数 `引数... -> 本体`（ブロック本体可）
  - 適用（引数はカンマ区切り・先頭から埋まる）、パイプ `|>`、実行 `!`
  - レコード構築（型名 + インデントのフィールド列 + `|>` 継続行。§2.7）
  - Vec リテラル（リスト形式 `-`。フィールド値・`=` 右辺。§2.8。リスト要素のレコードは型名なし）
  - `@ path` + インデントの束縛列挙、`:pub`（独立行）
  - `h` 要素構文・属性ペア・子ブロック・テキスト行・`{式}` 埋め込み（0-1〜0-4 の決定に従う）
- パーサ単体テスト（正常系 + 構文エラーの span 検証）。

### フェーズ4: 名前解決・スコープ検査

- 環境: prelude → import → ファイルトップレベル → 関数ローカルの階層。
- 暗黙捕獲禁止（0-7 の決定した線引きで検査）・シャドーイング全面禁止・再代入なし。
- 1定義=1実装（同一スコープから同じ `#` 定義の実装が2つ見えたらエラー）。
- 1ファイル=1export（`:pub` は1つまで）。
- `@ ./path` の解決: ファイル読み込み、列挙された束縛の存在検査、循環 import の検出。
- **フェーズ4 完了（2026-07-05）**。src/resolve.rs（Loader/Checker 分離、フレームのスタック＋捕獲境界の2軸）。実装中に発見した仕様の穴3件（`h` と同名の束縛・import パス解決規則・定義名の import 可否）はコア仕様 §10 の要決定 17〜19 に追記。トップレベルとブロックローカルの自己参照の非対称は要決定5に注記。`:pub` の型注釈必須はパーサ側で検査済みのため resolve では重複実装していない。

### フェーズ5: 型検査

- 単相。型: `i32 i64 f32 f64 String Bool Void Vec<T> Eff<T> Html Tag Document Recipe` + 関数型 + `#` 宣言由来のノミナルなレコード型。
- `#` 宣言・prelude シグネチャを型環境の起点とし、束縛は右辺から前進推論。HM 本実装はしない。
- `map` は intrinsic として適用点で要素型を特別扱い（`(A -> B), Vec<A> -> Vec<B>`）。
- html 検査: 未知の Tag（`.dvi` 等）＝エラー（近縁名のサジェストつき）。属性は要素ごとの許可表 + `data-*`/`aria-*` 任意許可。未知の属性＝エラー。`on-*` ＝エラー（js ターゲットの責務）。真偽属性は名前のみ。void 要素への子＝エラー。必須属性の欠落＝警告（html-spec §3.3 の【要決定】は格上げせず警告のまま）。
- レコード構築の全フィールド必須・宣言順・フィールドアクセスの存在検査。

## golden test 設計

- 1ケース＝1ディレクトリ: `tests/golden/<case>/main.clum`（+ 追加 `.clum`）。
- 正常系: `expected/` 以下に期待 `.html` を置き、ビルド結果とバイト一致で比較。
- エラー系: `expected_error.txt` に stderr 全文を置き完全一致で比較（エラーメッセージの品質が価値の中心のため、表示の退行も検出対象とする）。
- 環境変数 `UPDATE_GOLDEN=1` で期待値を再生成できるようにする。

## 使用モデルの選定（費用対効果）

### 料金（2026-06 時点、$/100万トークン）

- Claude Fable 5
  - 入力: $10.00 / 出力: $50.00
  - 特性: 最高性能。最難度の推論・長時間の自律作業向け
- Claude Opus 4.8
  - 入力: $5.00 / 出力: $25.00
  - 特性: コーディング・エージェント作業の標準。Opus 級最上位
- Claude Sonnet 5
  - 入力: $3.00（2026-08-31 まで $2.00） / 出力: $15.00（同 $10.00）
  - 特性: Opus に迫るコーディング性能。速度とコストの均衡点
- Claude Haiku 4.5
  - 入力: $1.00 / 出力: $5.00
  - 特性: 最速・最安。単純で機械的な作業向け

コスト比はおよそ Haiku : Sonnet : Opus : Fable = 1 : 3 : 5 : 10。
選定の原則: **手戻りのコスト > モデル差額なら1段上のモデルを使う**。安いモデルで2回
やり直すより、適切なモデルで1回で通すほうが総コストは低い。

### フェーズ別の推奨モデル

- 0. 仕様の穴の解決 → **Fable 5**
  - 言語設計の判断は全工程の土台で、誤りのコストが最大。仕様書間の整合性分析に最高の推論力を使う。作業量自体は小さく、総コストへの影響は軽微
- 1. 基盤 → **Sonnet 5**
  - CLI・span・golden ランナーは定型的。ただしエラー表示の設計は言語の価値の中心のため、表示形式の設計だけ Opus 4.8 でレビューする
- 2. 字句解析器 → **Sonnet 5**
  - INDENT/DEDENT 変換は既知のパターンで、仕様が明確・単体テストで担保できる
- 3. AST・パーサ → **Opus 4.8**
  - 文法全体の設計判断と span 規律が絡む中核。ここの手戻りが最も高くつく
- 4. スコープ検査 → **Opus 4.8**
  - 暗黙捕獲禁止は前例の少ない独自規則で、仕様解釈の繊細さが要る
- 5. 型検査 → **Opus 4.8**
  - 型検査の誤りはテストをすり抜けやすく、`map` intrinsic の特別扱いも設計判断を含む
- 6. 評価器 → **Sonnet 5**
  - 部分適用・レコード・パイプの意味論は仕様で確定済みで、golden test で検証できる
- 7. HTML 出力 → **Sonnet 5**
  - エスケープ・void・無挿入連結は機械的で明文化済み。emit 単体の実装は Haiku 4.5 でも可
- 8. golden 拡充・仕上げ → **Haiku 4.5**
  - ケースの量産・`fmt`/`clippy` 対応は機械的。期待値の妥当性判断を要するケース設計のみ Sonnet 5
