# 人間からの注文

ブログエンジンをclumで作りたい。
cloudflare workerにデプロイするが、デプロイ先は抽象化して後々別の(例えばecsのようなcontainerでもいけるように)サービスに引っ越しできるようにしたい。

ブログ自体はspa構成にしたい。
背景にwebgpuのアートを表示したい。
ページによって背景が変わる。変わる途中のトランジションもできるようにしたい。
※例えば水彩やインクアートのようなもの。

ブログのページを作るとき、そのコンテンツのhtmlだけを組みたい。

```html
<main>
  <h1>page1</h1>
  <article>
    <div>hello</div>
  </article>
</main>
```

訪問者が初回アクセスするときはcf workerが完全なhtmlとして返す。

```html
<!DOCTYPE html>
<html lang="ja">
  <head>...</head>
  <body>
    {page1}
  </body>
</html>
```

以後訪問者がリンクを押す度にpage1が切り替わる。
ページ内のscriptには以下の制約を持たせたい。
これはダミーなのでわかりやすいtypescriptで書くがclumでかけるようにしたい。

```ts
type Action<Deps> = {
  will(deps: Deps): Promise<void>
  did(deps: Deps): Promise<void>
}

type Page<Deps> = {
  open: Action<Deps>
  close: Action<Deps>
}
```

Deps はページによって異なる。
例えばrouter機能やwebgpu接続機能など。

```ts
build({
  route: '/aaa/bbb',
  element: component,
  dependencies: [
    Web.Router,
    Web.WebGPU,
  ],
})
```

１つのページが遷移するとき、routerやwebgpuへの参照を完全に消したい。
そのためにwillで参照を消す準備を、didで参照を消す処理を入れたい。
名前にはこだわりがないのでもっといい名前があればそっちにしたい。
UniRXの
```cs
Observable.SubScribe().AddTo(this);
```
のようなAddToも考えていたがよく忘れるケースがあった。
なのでwill/didの方が検知しやすいのでは？と考えている。

画像などのアセットは別のcdnにおけるようにしたい。

ページを作るときはviteのようなHMRに対応してほしい。
そういうサーバが欲しい。

LSPとシンタックスハイライトもほしい。
vscode/nvimでつかいたい。
シンタックスハイライトはwebページでも使いたい。

# 計画（2026-07-06 起案）

注文は layer 5（app: ブログ）の要求だが、実現には layer 1〜3 の大規模な拡張（js ターゲット・`Eff` 合成・ジェネリクス）とツールチェーン（dev server・LSP・ハイライト）が要る。
仕様が正、実装は従。各ステップは「要決定の解決 → 仕様書の改訂 → 実装」の順で進め、本計画では仕様の穴を**勝手に決めず選択肢の提示に留める**（§4）。

## 1. 注文の分解と現状（ステップ1完了時点）のギャップ

- ページをコンテンツ HTML だけで組む
  - 現状: **ほぼ可能**。ページを `Html` 値で書き、レイアウトをコンポーネント（`# Layout`）にし、同じ値から「完全版（layout 適用）」と「fragment 版（コンテンツのみ）」の2つの `Document` を出力すれば、SPA 用の断片も静的に得られる。言語拡張は原理上不要
  - ギャップ: レイアウトを別ファイルに置いた時点で定義名 import（コア §10-19）を踏む。ページ一覧から `Vec<Document>` を `map` で組めるか（コア §10-16 の範囲）も要確認
- 初回アクセスで完全な HTML を返す
  - 現状: ブログは全ページがビルド時に確定するため、**SSR は不要**。ビルド済み HTML を Worker が配信するだけで成立する
- リンククリックでページが切り替わる SPA
  - ギャップ: **js ターゲットが存在しない**。設計文書の未決（リアクティブ機構 §9-1、ターゲット間参照 §9-2、`on-*` の解釈 §9-3）の解決が前提
- `Page<Deps>` ライフサイクル（will/did）と Deps 注入
  - ギャップ: ジェネリクス（型パラメータつきレコード）・関数型フィールド・`Eff` の合成（コア §10-1）・Promise との対応（新規の穴）。ライフサイクルの語彙そのものは layer 4（lib）の設計であり、コアに手を入れずに成立させることが規約（デザイン §2-4）の試金石になる
  - 注文の動機（参照の消し忘れ検知）について: clum は不変束縛＋暗黙捕獲禁止のため「ページが参照を握りっぱなしにする」経路はスコープ規則で大きく塞がれている。漏れうるのは JS 側資源（イベントリスナ・GPU オブジェクト・購読）であり、ライフサイクル設計はそこに集中すればよい（§4 の選択肢参照）
- WebGPU 背景と遷移トランジション
  - ギャップ: `@js` interop（コア §8.1）の実装と WebGPU 語彙（layer 3/4）。シェーダ（WGSL）は clum の外のアセットとして扱う。トランジションは「旧ページ close → 背景遷移開始 → DOM 差し替え → 新ページ open」の順序制御であり、`!` の逐次実行モデルの上に素直に載る見込み
- デプロイ先の抽象化（Workers → 将来 container 等）
  - 方針: 出力は Web 標準の静的ファイル一式（原則6）なので、「静的配信＋パス規約」を契約とする薄いアダプタで足りる。アダプタは clum の外（手書きの数十行）に置く。将来のサーバターゲット検証（デザイン §10）の題材として温存
- アセットを別 CDN に置く
  - ギャップ: ベース URL の注入方法が未定義。当面は環境別の `main.clum`（`main.dev.clum` 相当を分ける）で言語変更なしに済ませる案を §4 に示す
- vite のような HMR
  - ギャップ: `clum serve` / `watch` が存在しない。段階導入する: 全再読み込み → fragment 差し替え（SPA 遷移経路の再利用）→ モジュール単位
- LSP・シンタックスハイライト（vscode / nvim / web）
  - ギャップ: すべて未着手。ただしコンパイラ前段（lexer〜typeck。span・診断つき）はそのまま流用できる資産

## 2. マイルストーン（提供価値の順）

- M1: 静的ブログ公開 — 言語拡張ほぼゼロで、ブログを Workers から配信する
- M2: 執筆体験 — serve（live reload）・ハイライト・LSP 診断
- M3: SPA 化 — js ターゲット＋ライフサイクル＋WebGPU 背景
- M4: 仕上げ — HMR 本実装・LSP フル機能・アダプタ2種目（container）

最大リスクは M3 を塞ぐ js ターゲットの仕様確定（§4 の B 群）。注文の中核（SPA・WebGPU・HMR 本実装）がすべてその上に載るため、依存しない M1・M2 を先に出して価値を確定させる。

## 3. トラックとステップ

### トラックA: ブログ（layer 4/5。clum で書く）

- A-1: 静的ブログ（M1）
  - 置き場所: 本リポジトリ内 `blog/`（dogfooding。言語側の修正と同じ変更単位で回せる。将来分離可）
  - スコープ: ページ数件・`# Layout` コンポーネント・完全版と fragment 版の2出力・素の `.css` アセット（css ターゲットは使わない）
  - 先に要る決定: 定義名 import（コア §10-19）。`'` を含む文字列（コア §10-13）は英文の属性値・補間で踏み次第エスカレーション
  - 完了条件: `clum build ./blog/main.clum` で全ページの完全版＋fragment 版が `dist/` に出る。golden に blog 相当の縮約ケースを追加
- A-2: Worker アダプタとデプロイ（M1）
  - アダプタ契約（デプロイ抽象の実体）:
    - `dist/` の静的ファイルを URL パスへ 1:1 で配信する
    - 拡張子なしパスは `<path>/index.html` へ解決する
    - SPA フォールバックは持たない（全ページが実 HTML を持つため 404 は本物の 404）
    - アセット URL はビルド時に CDN 側へ埋め込み済みで、アダプタは関知しない
  - この契約を満たす限り Workers / ECS+nginx / S3+CDN のどれでも同じ `dist/` が動く。Cloudflare 固有の知識はアダプタ内に閉じ込め、ブログ本体（clum ソース）からは一切参照しない
  - Worker 実装は clum の外の手書き JS（Static Assets 配信の最小構成）。リポジトリには `deploy/cloudflare/` として置く
  - 完了条件: 実 URL でブログが閲覧できる
- A-3: SPA 化（M3。トラックB完了後）
  - Router・ライフサイクル・WebGPU 背景を layer 4 の lib として clum で書き、ブログへ組み込む
  - 遷移列: リンク横取り → fragment を fetch → 旧ページ close（資源解放）→ DOM 差し替え → 新ページ open → 背景シェーダのトランジション
  - 完了条件: JS 無効でも全ページが文書として閲覧でき（原則: html 出力は自立）、JS 有効時は再読み込みなしで遷移し、遷移ごとに背景が変わる。DevTools で旧ページの リスナ・GPU 資源が残っていないことを確認する手順を検証に含める

### トラックB: 言語（js ターゲット）

- B-0: 仕様の要決定（着手ゲート。§4 の B 群を一括で解決し、コア仕様・HTML 仕様と対になる js 仕様書 `clum-js-spec.md` を起こす）
- B-1: js ターゲット最小
  - スコープ: `*.js.clum`（複合拡張子の解禁）→ ESM 出力、DOM 構築（`h` の js 解釈）、イベント（`on-*`）、`@js` interop、`Eff` 合成、`?` マッチ・直和型の解禁（ステップ1除外分の復帰）、ジェネリクスの必要範囲（HM 本実装の前倒し判断を含む）
  - 検証: 出力 JS の golden（バイト一致）に加え、挙動の e2e（ヘッドレスブラウザ）を導入するかは依存の相談（§5）とセットで判断
- B-2: SPA を支える語彙
  - スコープ: fetch・history・DOM 差し替えの `@js` バインディング、WebGPU の最小バインディング（device 初期化・パイプライン・uniform 更新・requestAnimationFrame）
  - ここは layer 3（target 語彙）と layer 4（lib）の境界を実地で引く工程。コアに手を入れずに成立しなければコア設計の欠陥として扱う（デザイン §2-4）

### トラックC: ツールチェーン（言語仕様に触れないため、トラックA/B と並行可）

- C-1: `clum serve ./main.clum`（M2）
  - 監視（mtime ポーリングで開始）→ 再ビルド → ブラウザへ再読み込み通知（SSE）。CLI 形式は既存規約どおり引数明示
  - std のみで実装できる見込み（TcpListener + HTTP/1.1 + SSE）。エラー時はビルド診断をブラウザ上にも表示する（エラーメッセージの品質は言語の価値の中心）
- C-2: シンタックスハイライト第1弾（M2）
  - web 向け: 自前 lexer を正としたビルド時ハイライト（コードブロックを静的な `<span>` 列に変換。JS 不要で原則6と整合）。ブログのコードブロックで dogfooding する
  - エディタ向け: TextMate 文法の粗版（vscode）。精密化は LSP セマンティックトークン（C-3）に委ねる
- C-3: LSP（M2 で診断まで、M4 でフル）
  - 段階: 診断（既存パイプラインの流用）→ ホバー型表示 → 定義ジャンプ → 補完（`Tag`・属性表は閉集合なので補完と相性が良い）→ セマンティックトークン → リネーム
  - vscode 拡張と nvim（組み込み LSP クライアント）の設定例を同梱する
- C-4: HMR 本実装（M4。トラックB完了後）
  - 段階2: fragment 差し替え — 編集されたページの fragment を再ビルドし、SPA の遷移経路（close → 差し替え → open）をそのまま再利用して注入する。専用機構を増やさない
  - 段階3: js モジュール単位の差し替え — js ターゲットのモジュール境界に依存するため、B-0 の仕様設計時に「HMR 境界」を制約として織り込むかを判断する

## 4. 仕様の要決定（人間の判断を仰ぐ一覧）

M1 ゲート（トラックA-1 を塞ぐ）:

- 定義名 import（コア §10-19） — レイアウト分割に必須。選択肢は §10-19 記載の (a) 値 import が `#` 宣言を随伴 / (b) `#` 宣言も `:pub` 対象 / (c) 定義名 import を持たない
- アセットの扱い — 選択肢: (a) 当面ビルド外（デプロイ工程がコピー。言語変更なし） / (b) `Recipe` にコピー宣言を追加（仕様改訂）。提案は (a)
- CDN ベース URL の注入 — 選択肢: (a) 環境別 `main.clum` を分ける（言語変更なし） / (b) `clum build` にビルド時定義の注入機構を足す。提案は (a)

M3 ゲート（トラックB-0。js 仕様書の起草として一括で扱う）:

- js リアクティブ機構（デザイン §9-1） — クロージャのみか signal か。SPA の再描画単位に直結
- ターゲット間参照（デザイン §9-2） — js から html のマウントポイントを取る方法
- `Eff` 合成（コア §10-1） — `~` / `<-` の割り当て。ライフサイクルの逐次実行に必須
- 非同期と `Eff` の対応（新規の穴） — JS の Promise を `Eff<T>` に畳み込むか、`Async<T>` を分けるか。fetch・GPU 初期化・トランジション完了待ちのすべてが踏む
- `?` マッチ・直和型の解禁 — ルーティング分岐に必要（ステップ1除外分の復帰範囲）
- ジェネリクスと制約（コア §10-3 含む） — `Page<Deps>` の型付け。HM 本実装をこのステップで行うか
- `Stream<T>` の最小演算（コア §10-5 と対） — イベント列・requestAnimationFrame の表現。一般再帰禁止の帰結として必須
- `on-*` と属性値の区別（デザイン §9-3）
- `Recipe`/`Document` の拡張 — JS 出力の宣言方法（`Document` は現在 `element: Html` のみ）。複合拡張子 `*.js.clum` の解禁と対
- 1ファイル＝1export と index 規約（コア §10-6） — Router 等の lib を書くと即踏む
- ライフサイクルの形と命名（layer 4 設計だが型の表現力に跳ねるため先に方向を決める） — 選択肢:
  - (a) 注文どおり `open`/`close` × `will`/`did` の4フック
  - (b) `open` が「解放の Eff」を返す（`# Release` を `Eff<Void>` の別名とし `open: Deps -> Eff<Release>`。解放を書き忘れると型が合わず、router が close 時に必ず実行する）
  - (c) UniRx の AddTo 相当（登録先を明示的に渡す）
  - 検討の観点: 注文の動機は「消し忘れの検知」であり、(b) は忘れそのものを型エラーにできる。4フックが要る場面（トランジションの準備と確定の分離）が実在するかで (a) との併用も含め判断

ツールチェーン（トラックC）:

- ハイライトの供給源 — 選択肢: (a) TextMate と tree-sitter を別々に手書き（web は Shiki） / (b) 自前 lexer を正とし web はビルド時ハイライト、エディタは LSP セマンティックトークン主体で TextMate は粗版のみ / (c) tree-sitter を正とし web は wasm。提案は (b)（文法定義の三重保守を避け、原則6とも整合）
- HMR の段階3（モジュール単位）を目標に含めるか — 段階2（fragment 差し替え）で執筆体験としては十分な可能性がある

## 5. 依存クレートの相談ポイント（規約: 追加は事前相談）

- C-1 serve: std のみで実装可の見込み（TcpListener・SSE・mtime ポーリング）。相談不要で開始し、限界が見えたら相談
- C-3 LSP: JSON-RPC と JSON の実装が要る。候補: (a) std のみ手書き（JSON パーサ自作） / (b) serde + serde_json + lsp-types（デファクト・薄い） / (c) tower-lsp + tokio（厚いが枯れている）。着手時に理由と代替案を添えて相談
- B-1 の e2e 検証: ヘッドレスブラウザ駆動（例: 外部プロセスとして chromedriver）を導入するか。golden（出力 JS のバイト一致）だけで開始し、挙動検証の必要が出た時点で相談
- Worker アダプタ（clum の外）は wrangler 等の Node ツールを使うが、Rust 本体の依存には影響しない

## 6. やらないこと（先取り禁止の再確認）

- css ターゲット — 当面は素の `.css` アセット。ブログのスタイルが安定してから題材にする
- wasm ターゲット・サーバターゲット — Worker アダプタは clum の外の暫定実装に留める
- リアクティブ機構の実装先行 — 仕様（B-0）の決定前に signal 等を書き始めない
- markdown 入力・raw HTML 出力・HTML コメント出力 — 注文に含まれず、既存の要決定（html 仕様 §7-4・§7-8）を実装の都合で先に決めない
