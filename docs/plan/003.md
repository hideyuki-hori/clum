# ブログエンジン計画（エンジン集中）

対象はエンジン本体のみ。LSP・シンタックスハイライトはエンジンではないため本計画に含めない（別計画とする）。

## ゴール

- 執筆者はページの**コンテンツ HTML だけ**を clum で書く（`<main>...` 相当の断片）
- 執筆は生の tag（`h .div` 等）ではなく、**ライブラリ提供のコンポーネント**で書く。markdown に似た pug 風の書き味で、HTML のセマンティクスは lib 側が決める。執筆中はコンテンツだけに集中できる
- 初回アクセスはレイアウト適用済みの**完全な HTML** を返す
- 以降のリンククリックは fragment 差し替えの **SPA 遷移**
- 背景に **WebGPU アート**。ページごとに変わり、遷移トランジションを持つ
- 遷移時、旧ページの router / webgpu への参照は**完全に解放**される（ライフサイクルで検知可能に）
- デプロイ先は抽象化（まず Cloudflare Worker、将来 container 等へ引っ越し可）
- アセットは別 CDN に置ける
- vite 的な HMR を持つ dev server

## エンジンの6部品

- 執筆コンポーネント lib — `Para` `Section` `Code` 等、執筆に使う語彙を layer 4 の lib として clum で書く。HTML セマンティクスへの対応（`Para` → `<p>` 等）はこの lib に閉じ込め、生の `h` は lib とレイアウトの中にだけ現れる。言語拡張は不要（タグ位置の UpperCamelCase 解決・裸のテキスト行・無挿入連結が既にある）。文中リンク・強調は行分割＋無挿入連結で表現する
- ビルダ — ページ（コンテンツ Html）にレイアウトを適用し、**完全版と fragment 版の2出力**を得る。既存の `Recipe`/`Document` で表現できる（ルートが `h .html` でなければ fragment 出力になる規則は実装済み）
- 配信アダプタ — 「`dist/` を URL パスへ 1:1 で静的配信する」だけの薄い契約。この契約に依存する限り Worker / container / S3 のどれでも同じ `dist/` が動く。実装は clum の外（`deploy/cloudflare/` の手書き JS 数十行）
- クライアントランタイム — リンク横取り → fragment を fetch → 旧ページ close → DOM 差し替え → 新ページ open。clum の js ターゲットで書く（**js ターゲットは未実装**。ここが山場）
- ライフサイクル — `Page<Deps>` と `open`/`close` × `will`/`did`。layer 4（lib）として clum で書き、コアに手を入れずに成立させる
- 背景エンジン — WebGPU の最小バインディング（device・パイプライン・uniform・rAF）。シェーダ（WGSL）は clum の外のアセット

## フェーズ（提供価値の順）

- P1: 静的エンジン — 現行言語（ステップ1）でビルダと執筆 lib を成立させる
  - `blog/` にページ数件＋レイアウト＋執筆コンポーネント lib。ページは執筆 lib のコンポーネントとテキスト行だけで書く。全ページの完全版＋fragment 版が `clum build` で出る
  - 仕様の穴は 2026-07-12 に解決済み（下記「P1 ゲートの決定」）。言語側の実装が先行タスクになる: `:pub`・1ファイル1export の廃止、`^名前` 行、役割ファイル（窓口 `_.clum`。2026-07-18 統一）、ディレクトリ限定 import（`@.` 含む）と可視性検査、双方向随伴、CLI のディレクトリ指定化（`clum build ./main.clum` 形式の廃止）、examples・golden の追随更新
- P2: 配信 — アダプタ契約の確定と Worker 実装。実 URL でブログが見える。アセットはビルド外コピーで開始
- P3: js ターゲットの仕様確定 — `clum-js-spec.md` の起草。`Eff` 合成・Promise との対応・ジェネリクス（`Page<Deps>`）・`on-*`・ターゲット間参照・`?` マッチの解禁範囲。**起草 → 人間レビュー → 実装**の順を厳守
- P4: SPA 化 — js ターゲット実装 → Router・ライフサイクル lib → ブログ組み込み。JS 無効でも全ページが文書として読めることを維持
- P5: WebGPU 背景 — バインディング＋ページごとのシェーダ切替＋トランジション（close → 背景遷移 → 差し替え → open の順序制御。`!` の逐次実行モデルに載せる）
- P6: dev server — watch → 再ビルド → reload から始め、fragment 差し替え HMR（P4 の遷移経路を再利用）へ

## P1 ゲートの決定（2026-07-12 解決）

- 定義名 import — 双方向随伴で解決（定義名 `Card` 1つで `#` 宣言と実装が運ばれる。コア仕様 要決定19・30）
- 多数コンポーネントの公開 — `:pub` と「1ファイル＝1export」を廃止し、ファイル密閉＋`^` ブロック＋ディレクトリ窓口の2段宣言に全面改訂（コア仕様 §2.3・§8.2・§8.4、要決定6）
- あわせて確定: エントリポイントは役割ファイル `-.clum`（`main.clum` は廃止。当初案 `!.clum` はシェルの履歴展開で素打ちエラーになるため改名）・import の対象はディレクトリのみ（`@.` と `@パス`。境界の守りは可視性検査）・CLI は `clum build <ディレクトリ>`（引数なし）・環境差分はディレクトリで分け、値はソースに直書きする（ビルドはソースツリーのみから決まる）
- 2026-07-18 追記: 役割ファイルは窓口 `_.clum` の1種へ統一（`-.clum`・`^.clum` を廃止。1ファイルが公開面と式列の両方を担う。コア仕様 §8.4）
- 残る要決定は31（複数段の下り import のパス規則）のみ。P1 のソースは1段の import だけで書けるため先送り可

P3 以降の要決定（`Eff` 合成の記号、ライフサイクルの形と命名等）は P3 の仕様起草時にまとめて提示する。先に決めない。
