# 2リポジトリ体制への再編（plan/003 を置換）

2026-07-18 の方針決定（Claude.ai 議論からの引き継ぎ）を受け、ブログエンジン計画（plan/003）を clum / every.fail の2リポジトリへ割り直す。

## 前提の決定（2026-07-18）

- clum に求めるのは書き味。速度・複雑な処理・ランタイムは Rust / JS 等の実装へ委譲する
- フレーミングは「YAML の進化版」: 型検査・不変束縛・モジュールと可視性・合成を持つデータ記述言語。残すコアは「構造と名前の管理」（レコード・Vec・コンポーネント・モジュール境界・双方向随伴）
- 領域の線引きは app / lib の縦ではなく、構造（clum）/ 計算（Rust・JS）の横で切る。設計の重心は言語本体から境界のインターフェースへ移る（`Recipe` / `Document` / `build` が境界契約の第1号）
- リポジトリのトップは責務で切り、実装言語は各パッケージの内部事情に沈める。パッケージ置き場は `boundaries/`
- ブログは every.fail リポジトリに置き、そこから clum を使う（dogfooding 第1号）。clum リポジトリは境界を提供する側に徹する

## 責務分担

- clum リポジトリ — 言語 core・CLI・境界パッケージ（配信アダプタ契約・dom・クライアントランタイム・WebGPU バインディング）・examples・golden test
- every.fail リポジトリ — 執筆コンポーネント lib・レイアウト・ページ・WebGPU シェーダ・サイト固有のすべて

## フェーズ

- P1: 言語実装の追随（clum のみ。**every.fail の作業はスコープ外**・2026-07-18 決定）
  - `:pub`・1ファイル1export の廃止、`^名前` 行 export
  - 窓口 `_.clum`（2026-07-18 統一形: 公開面＋式列。import された窓口の式列は実行しない）
  - ディレクトリ限定 import（`@.` 含む）と可視性検査（関所の連鎖）、双方向随伴
  - CLI `clum build <ディレクトリ>`（ファイルパス形式の廃止）
  - examples・golden の新形式への追随
  - 完了条件: latest.md「実装の追随状況」が空になる
- P2: 構成移行 — `src/` → `boundaries/core`（lib）＋ `boundaries/cli`（bin。パッケージ名は `clum` を維持）
  - crate 分割はこの2つから始める。html-renderer 等の切り出しは境界が実際に必要になった時点で行う
  - CI は言語ではなくパスフィルタで切る。各 boundary 直下に契約（公開インターフェース）と実装言語を1枚書く
- P3: every.fail 立ち上げ — 配布と破壊検知の確定
  - 配布: core（CLI）は `cargo install --git`。every.fail 側は rev / タグ固定で参照する
  - 破壊検知: every.fail の通常 CI は固定 rev でビルドし、別途 clum main を取ってビルドする scheduled ジョブで通知のみ行う（clum 本体の CI は重くしない）
- P4: 配信（旧 P2） — アダプタ契約（`dist/` を URL パスへ 1:1 で静的配信する薄い契約）の確定は clum 側、Worker 実装と実サイトは every.fail 側
- P5: js ターゲットの契約起草（旧 P3） — 「clum で JS を書く仕様」ではなく「clum の構造宣言を JS ランタイムがどう解釈するかの契約」として `clum-js-spec.md` を起草する。起草 → 人間レビュー → 実装の順を厳守
- P6: SPA 化（旧 P4） — dom・クライアントランタイム・ライフサイクルの境界パッケージは clum 側、ブログへの組み込みは every.fail 側。JS 無効でも全ページが文書として読めることを維持
- P7: WebGPU 背景（旧 P5） — バインディングは clum 側、シェーダ（WGSL）とページごとの適用・トランジションは every.fail 側
- P8: dev server（旧 P6） — watch → 再ビルド → reload から始め、fragment 差し替え HMR（P6 の遷移経路を再利用）へ

## やらないこと

- P1 での every.fail 側作業の先取り（執筆 lib・レイアウト・ページは every.fail 側の仕事）
- LSP（別計画のまま。シンタックスハイライトは作成済み: `editors/` の nvim・vscode）
- css / wasm ターゲットの先取り、仕様決定前のリアクティブ機構・signal の実装先行
- npm 公開（時期尚早。JS 側 boundary の参照方法は P6 が近づいた時点で決定する）
