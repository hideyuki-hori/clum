# clum コア仕様ドラフト v1

コア層＝全ターゲット（html / css / js / 将来の追加ターゲット）が共有する部分。
ここに含まれるのは **字句・構文・型システム・スコープ・実行モデル・モジュール** のみ。
継ぎ足しても壊れないための規約（言語デザイン文書 §2）により、コアは凍結に近い扱いとする。ここでの決定がすべての土台になる。

**レイヤー構造上の位置づけ**: このドキュメントは 5 層構造（layer 1: core 〜 layer 5: app）のうち layer 1（core）と layer 2（prelude）の両方を扱う。`:is` `:any` `:else` `:share` の4語（§1.6。`:out` は 2026-07-04 に、`:pub` は 2026-07-12 に廃止）はモジュール・型・スコープの意味論に直結するため layer 1 側に置くが、§2.4 の `# :name` 機構を使って定義される具体的な述語・制約（`:lt`, `:comparable` 等）は layer 2（prelude）に属する。層の全体像は言語デザイン文書 §2 を参照。

【要決定】は提案であり未確定。

## 1. 字句

### 1.1 ソース

- UTF-8。
- インデントはスペースのみ。1レベル＝2スペース固定（決定済み）。
- タブはソースのどこにあってもコンパイルエラー（決定済み・2026-07-05）。例外は html のテキスト行の中身と文字列リテラルの中身のみ（生データとして保持される）。
- コメントは `//` のみ。ブロックコメントは持たない。

### 1.2 識別子

- 定義: UpperCamelCase（例: `Card`, `SaveItem`, `Add`）
- 変数・関数: lower-kebab-case（例: `current-target`, `save-item`）

`:` 呼び出し（§2.4）は独立した命名カテゴリではありません。`:add` は定義名 `Add` を lower-kebab-case にした綴りであり、新しい識別子ではなく既存の定義への別の参照方法です。

### 1.3 kebab-case

`-` は識別子の一部としてのみ現れます。演算子としての `-`（減算）はコアに存在しないため（§1.5, §2.6）、`a-b` は空白の有無にかかわらず常に識別子であり、曖昧性は生じません。

### 1.4 リテラル

- 整数: `42`, `-1`（既定型 `i32`）。負号はリテラルの字句の一部であり、演算子ではない（§1.3, §2.6）。
- 小数: `3.14`（既定型 `f64`）
- 文字列: `'...'` のみ。`"` の文字列構文は持たない（1機能1記法）。
- 補間: 文字列内 `{式}`。リテラルの `{` `}` は `{{` `}}` と書く（単独の `}` もエラー。決定済み・2026-07-05。html のテキスト行の規則と同一の対称形）。
- 文字列リテラルは1行内に収める（改行を含む文字列の表記は持たない。決定済み・2026-07-05）。`'` 自体を文字列に含める手段は【要決定】（§10）。
- 文字リテラルは存在しない（決定済み）。`char` 型自体を初期セットに含めず、文字列は `String` のみで扱う。文字単位の処理は Unicode の単位（コードポイント／書記素）の決定を伴うため、必要になった時点で edition 単位で導入を検討する。

### 1.5 記号の割り当て

- `#` — 定義の宣言
- `@` — import
- `?` — パターンマッチ
- `!` — 効果の実行
- `|>` — パイプ
- `->` — 関数・マッチアームの帰結（決定済み: `=>` は存在しない）
- `^` — export（定義に密着前置して兄弟可視を宣言する。`@` の対。§8.2。2026-07-12 決定・2026-07-18 第2次改訂で定義前置へ）。窓口では `@` と合成した `^@パス` ブロックがディレクトリの外への公開を宣言する（§8.4）
- `_` — 無名（2026-07-18 決定）。役割ファイル名 `_.clum`（§8.4）専用の印で、言語の字句には現れない（名を持つものは kebab-case / UpperCamelCase で書き、名を持たないものにだけ `_` が現れる）
- `<-` — 【要決定: ブロック脱出ガード or Eff 束縛】
- `~` — 【要決定: Eff 束縛 or 未割り当て予約】

**`+` `-` `*` `/` はコアの記号に含まれません。** 四則演算は §2.6 の修飾子（`:add` `:sub` `:mul` `:div`）として prelude（layer 2）が提供します。これにより §1.3 の kebab-case とのあいだの曖昧性そのものが発生しなくなります。

### 1.6 記号と `:xx` の二層構造

拡張性の入り口を1箇所に固定する。

- **記号はコアの専有物。** 上表の記号のみが存在し、ユーザー・ライブラリによる演算子定義は一切できない。新記号の追加はコア変更であり、edition 境界でのみ検討する。
- **`:` 呼び出しは拡張可能。** 通常の `#` 定義（layer 2 が供給する）は、シグネチャの形が合えば `:` 付きの綴りでも呼べます（§2.4）。新しい識別子ではなく、既存の定義名を lower-kebab-case にした綴りにすぎません。
- コア予約4語（`:any` `:is` `:else` `:share`。`:pub` は 2026-07-12 に廃止し、export は記号 `^` へ移した）のうち、実際に layer 1 に残るものが何かは §2.4 の各論・以降の検討事項です。layer 1 に残るものは、対応する `#` 定義を持たない、注釈専用の綴りとして扱われます。

「記号＝凍結、コロン語＝ライブラリで成長」という分担が、継ぎ足しても壊れないための規約（言語デザイン文書 §2）の字句レベルでの実体となる。

### 1.7 ドットの二つの用法（決定済み・2026-07-04）

`.` の役割は、前後の空白の有無だけで字句レベルで一意に決まる。

- **フィールドアクセス**: `識別子.識別子`（`.` の前後に空白を挟まない）。`page.title` はレコード `page` のフィールド `title` の読み出し。連鎖できる（`e.current-target.value`）。
- **ドット前置リテラル**: 空白のあとに置く `.識別子`。`h .div` の `.div`（`Tag`）、`:share .all` の `.all` のように、閉じた集合の1要素を指す綴り。

## 2. 定義と束縛

### 2.1 `#` 宣言

`#` は名前に**型（シグネチャ）を与える**宣言。

```
# User name: String, age: i32          // 積型（レコード）
# SaveItem api: Api, item: Item -> Void  // 関数シグネチャ
# Session :any LoggedIn User, Guest      // 直和型
```

### 2.2 実装と束縛

```
save-item: SaveItem api, item ->
  ...

x = 1 :add 2       // 束縛。再代入は存在しない
```

- すべての束縛は不変。`mut` に相当する概念はコアに持たない。
- 束縛には型注釈を付けられる: `x: T = 式`（2026-07-04 決定）。注釈が必須なのは境界（`^` に列挙された定義等、§3.1）のみ。
- **1定義＝1実装**: 同一スコープから同じ定義の実装が2つ見える状態はコンパイルエラー。コンポーネントタグの定義名解決（HTML 仕様 §2.2）はこの規則に依存する。

### 2.3 可視性の3段（2026-07-12 全面改訂・2026-07-18 第2次改訂）

可視性は「ファイル私有 → 兄弟可視 → ディレクトリの外」の3段で、境界を越えるたびに明示の宣言が要る。

- **`^` なしのトップレベル束縛・`#` 宣言はファイル私有。** ファイルの外のどこからも見えない。
- **`^` を定義に前置するとファイルの公開面（§8.2）。** 同じディレクトリの兄弟ファイルが `@./ファイル名` で明示 import して使う（§8.1。2026-07-18 第3次改訂）。ファイルの住所が通用するのは同じフォルダの中だけで、名前の出どころは使う側のファイルの import 列に必ず現れる。
- **ディレクトリの外へは窓口 `_.clum` の `^@` ブロック（§8.4）。** 列挙された名前だけが外から見える。index ファイル特例（特定名のファイルにだけ公開を許す緩和）は採らない（要決定6の解決）。

### 2.4 `:` 呼び出し（コアが与える唯一の拡張点）

`:` は**宣言の記法ではなく、呼び出しの記法**でございます。`#` 宣言そのものは何も変わりません。通常の `# Name` / `name: Name ...`（UpperCamelCase の型名と lower-kebab-case の実装名という既存パターン）のままです。

```
# Is a: T, b: T -> Bool
is: Is a, b -> ...
```

このとき、`Is` はシグネチャの**形**に応じて、通常の呼び出し（`Is a, b`）に加えて `:` 付きの呼び出しでも参照できます。`:` に続く綴りは**定義名を lower-kebab-case にしたもの**です（`Is` → `:is`）。

- `..., Self -> Bool` の形（対象が末尾引数）→ `?` のアーム位置で `:is 引数...` として使えます（**述語**、§3.3）。
- `Self` を含む複数メンバーの束 → 型パラメータの境界位置で `<T: is>` として使えます（**制約**、§3.4）。
- メンバーが1つだけの制約 → 中置記法 `a :is b` としても使えます（§2.5）。
- どの形にも当てはまらない定義（`# SaveItem` 等）は `:` 付きでは呼べません。通常の呼び出しのみです。

```
? session
  is Guest -> ...      // 通常呼び出し。コンストラクタ名そのものでも束縛できる（§3.3）
```

- `Self` は `#` 宣言の本体内でのみ使える組み込みの型変数で、「呼び出し側で解決される型」を指します。新しい記号は導入せず、コアが予約する識別子として扱います。
- **命名以外に新しい構文は一切ございません。** `#` 宣言の書き方は1種類のままで、`:` はその定義を呼ぶときの、もう一つの綴りにすぎません。以前の案（宣言名に `:` を前置する、`Modifier` という型を新設する）はいずれも撤回し、この呼び出し規則1つに統合いたします。
- コア予約の4語（`:any` `:is` `:else` `:share` のうち、実際に降ろせるものは §1.6 参照）と同名の定義は、同一スコープで両方が見える場合にコンパイルエラーです。
- 【要決定】この規則は「形が合えば自動的に `:` 呼び出しが有効になる」という**暗黙の**振る舞いです。`# IsAdult age: i32 -> Bool` のような、パターンマッチでの使用を意図していない普通の真偽値関数まで自動的に `:is-adult` として呼べてしまう点は、「暗黙の何かを認めない」というコアの精神（§6.1 の暗黙捕獲禁止と同種の懸念）と緊張関係にあります。何らかの明示的な opt-in（例: シグネチャ内の注釈）を要求するかどうかは未確定です。

この一段階の一般化により、§1.6 で述べた「記号＝凍結、コロン語＝ライブラリで成長」という分担を、新規構文を一切増やさずに実現いたします。ライブラリが行うのはすべて既存の `#` 宣言の範囲内の操作でございます。

### 2.5 用途3: 中置記法

§2.4 で述べた「述語」「制約」に加え、制約のうち**メンバーがちょうど1つ**のものは、追加のキーワードなしに中置記法でも使えます。

```
# Add
  add: Self, Self -> Self

x = 1 :add 2
```

- `a :name b` は、`a` の型に対する `Name` の実装が持つ唯一のメンバーの呼び出しへと脱糖されます。
- メンバーが2つ以上ある制約（例: `Comparable` が将来 `compare` 以外のメンバーを持つ場合）は中置記法の対象外で、通常どおり定義名参照（`Comparable.compare a, b` のように、モジュール等の別の場面で使う参照構文を流用）で呼び出します。
- 用途の判定基準がここでも**シグネチャの形のみ**である点は §2.4 と変わりません。中置記法専用の宣言キーワードは導入いたしません。

### 2.6 四則演算はコアの記号ではなく prelude の定義

`+` `-` `*` `/` はコアの記号表（§1.5）に存在いたしません。四則演算は §2.4/§2.5 の仕組みだけを使い、prelude（layer 2）が単一メンバーの制約として、通常の `#` 宣言で提供いたします。

```
# Add
  add: Self, Self -> Self

# Sub
  sub: Self, Self -> Self

# Mul
  mul: Self, Self -> Self

# Div
  div: Self, Self -> Self
```

```
x = 1 :add 2
y = x :mul 3
```

- `i32` `i64` `f32` `f64` は prelude によってこれら4制約の実装を持つ状態で出荷されます。実装そのものは処理系組み込み（intrinsic）であり、`# Add` はその**インターフェース**を宣言しているにすぎません（§3.3 の `Lt` と同様の位置づけです）。
- **既存の演算子方式に対する利点は拡張性です。** `+` `-` `*` `/` が記号として固定されていた場合、独自の数値型（`Vector`, `Money` 等）に対する演算子オーバーロードには専用の言語機構が別途必要になります。この方式であれば、任意の型が同じ `# Add` を実装するだけで `v1 :add v2` が成立し、コアに何も追加いたしません。
- 【要決定】除算は 0 除算を伴い得るため、`div: Self, Self -> Self` のまま（未定義動作／panic 相当）にするか、`div: Self, Self -> Eff<Self>` として失敗を効果として明示するかは未確定です。後者を選ぶ場合、`x :div y` は `Eff<Self>` を返すため `!` を要求する形になり、§5（実行モデル）との整合性が高くなります。
- 比較（`<` `>` `==` 等）についても同様の扱いとするかは別途要決定です。すでに §3.3 で `Lt` を述語の例として使用しておりますので、四則演算と対称的に「比較は述語シグネチャの `#` 定義、算術は単一メンバー制約の `#` 定義」という整理が自然に成り立ちます。

### 2.7 レコード構築（決定済み）

積型（レコード）の値は、**型名（UpperCamelCase）＋インデントしたフィールド列**で構築します。宣言（§2.1）と同じ見た目を構築にも使う、という対応です。

```
# Recipe documents: Vec<Document>

Recipe
  documents:
    - path: './dist/index.html'
      element: index
    - path: './dist/about.html'
      element: about
  |> build
  |> !
```

（例は 2026-07-04 の第2次改訂後の形。`documents:` の値はリスト形式の Vec リテラル、§2.8。）

- フィールドは1行1つ、`名前: 値` の形。カンマは書きません（行がフィールドの区切りです）。
- 全フィールド必須。省略・既定値の仕組みは持ちません（曖昧性排除）。順序は宣言と同一であることを要求します。【要決定: 順序自由を許すか】
- **`|>` による継続**: 構築ブロック内で `|>` から始まる行は、フィールド列の終わりを示し、**構築されたレコード全体**を右辺へ流します。フィールド行は必ず `識別子:` で始まるため、`|>` 行との区別は字句だけで一意に決まります。上の例は `build recipe |> !` と同じ意味であり、`Recipe` の構築 → `build` へパイプ → `!` で実行、と読み下せます。
- インラインの構築記法（`Recipe { documents: ... }` のような1行形式）は**持ちません**。1機能1記法です。
- 構築は束縛の右辺にも書けます。`x = Recipe` のように行の途中から構築を始めた場合、フィールド列は**その行より1レベル深いインデント**で続きます（HTML 仕様の `index = h .html` と同じ規則。2026-07-04 決定の帰結）。

### 2.8 Vec リテラル（2026-07-04 改訂: リスト形式に一本化）

`Vec<T>` の値は、`-` で始まる行を縦に並べた**リスト形式**で書きます（LiveScript / YAML のリストに倣う）。

```
xs =
  - 1
  - 2
  - 3

documents:
  - path: './dist/index.html'
    element: index
```

- 各要素は `- `（ハイフン＋空白。2文字＝インデント1レベル分）で始まります。
- 要素が**レコード**の場合、型名は書かず、最初のフィールドを `- ` と同じ行に、残りのフィールドを最初のフィールドと同じ桁（`-` から1レベル深い位置）に揃えて書きます。要素型はリストの期待型から一意に定まるため、型名の省略に曖昧性はありません。単独の式の位置でのレコード構築は従来どおり §2.7 の型名つきの形であり、場面が重ならないため1機能1記法は保たれます。
- 要素が**1行に収まる式**（リテラル・識別子参照等）の場合は `- 式` を1行で書きます。複数行にわたる式（`h` 式等）を要素に直接書けるかは【要決定】。必要な場合は先に `=` で束縛し、識別子で参照します。
- リストを書ける位置は、**レコードのフィールド値**（`名前:` の直後で改行し、1レベル深く置く）と **`=` 束縛の右辺**（`=` の直後で改行し、1レベル深く置く）です。
- **旧仕様のカンマ区切り形式（`files: index, about` / `xs = 1, 2, 3`）は廃止**（2026-07-04）。「期待型が `Vec<T>` の位置でのみカンマ列を Vec と解釈する」という文脈依存の解釈規則そのものが不要になり、カンマの意味は関数適用の引数区切りだけになります。
- 【要決定: 空の `Vec` の表記】は引き続き未決。

## 3. 型システム

### 3.1 基本方針

- 静的型付け、Hindley–Milner 型推論。
- 型注釈が必須なのは**境界のみ**: `^` に列挙された定義（2026-07-12。旧 `:pub`）、JS interop、ターゲット間参照。内部は推論に任せる。

### 3.2 組み込み型

```
i32  i64  f32  f64  String  Bool  Void
Vec<T>
Eff<T>        // 未実行の効果（§5）
```

- 数値型は最初から `i32` `i64` `f32` `f64` の4種（決定済み）。WASM の値型との1対1対応を維持するため（言語デザイン文書 §10）。
- `char` は初期セットに含めない（決定済み、§1.4）。

### 3.3 直和型とマッチ

```
# Session :any LoggedIn User, Guest

? session
  :is LoggedIn user -> ...   // ペイロードの束縛
  :is Guest        -> ...
  :else            -> ...
```

- マッチは網羅性検査あり。全ケースを尽くしている場合 `:else` は書けない（到達不能はエラー）。
- `?` は式であり値を返す。全アームの型は一致しなければならない。

#### 述語パターン（`:` 呼び出しの応用、§2.4）

`:is`（コンストラクタマッチ。実体は §3.3 冒頭のとおりコンストラクタ名の裸の参照）に加え、アーム位置の `:name 引数...` は**述語**として解釈されます。述語は §2.4 のとおり、シグネチャが `..., Self -> Bool` の形をした通常の `#` 定義です。

```
# Lt threshold: i32, Self -> Bool
lt: Lt threshold, target -> ...
  // i32/i64/f32/f64 に対しては処理系組み込みの比較で実装される（intrinsic）

? n
  :lt 0 -> 'negative'
  :else -> 'positive'
```

- `Lt` のようにプリミティブ数値型を対象とする述語は、宣言（インターフェース）は通常の `#` で書かれますが、`i32` 等に対する実際の比較は言語処理系が直接提供する組み込み実装（intrinsic）です。ユーザー定義の述語がさらに別の述語を呼び出して合成することは可能ですが、どこかで必ず組み込み実装に行き着きます（無限後退を避けるための境界です）。
- コアが持つのは §2.4 の呼び出し規則のみで、`Lt` `Gt` `In` のような個々の語彙はすべてライブラリが通常の `#` と実装で供給いたします。`@` で import すれば使えます。
- コア予約のコロン語（§1.6 の4語のうち layer 1 に残るもの）と同名の定義は、同一スコープで両方が見える場合にコンパイルエラーです。
- 網羅性: 述語アームは網羅性検査に**寄与しません**。述語アームを1つでも含むマッチは、コンストラクタで全ケースを尽くすか `:else` を置くかのどちらかが必須です。

#### ブロック脱出ガード 【要決定】

逐次処理の途中に置く1行マッチ。成立時はその値で**現在のインデントブロックを脱出**し、不成立時は次の行へ続く。

```
n = 1 :add 2
n ? :lt 0 <- 'invalid'
...続きの処理
```

脱糖規則（後続行が暗黙の `:else` アームになる）:

```
? n
  :lt 0 -> 'invalid'
  :else ->
    ...続きの処理
```

- 脱出先は関数ではなく**最も内側のインデントブロック**。ラムダ越しに外へ跳ばないため、early return の古典的な曖昧さを持たない。
- ガードの値と、ブロックの通常の最終値は同じ型でなければならない（脱糖後の既存規則から従う）。
- 未決: 記号を `<-` にするか（`Eff` 束縛との取り合い、§1.5）。

### 3.4 ジェネリクス

- `<T>` による型パラメータ。`# Map<A, B> f: (A -> B), xs: Vec<A> -> Vec<B>`
- 型クラス相当の制約は §2.4 のとおり、`Self` を含む複数メンバーの束を持つ通常の `#` 定義で表現します。新しい構文（`:constraint` のような専用キーワード）は導入しません。

```
# Comparable
  compare: Self, Self -> i32

# Max<T: Comparable> a: T, b: T -> T
max: Max a, b ->
  ? Comparable.compare a, b
    :lt 0 -> b
    :else -> a
```

- 型パラメータの境界は `<T: Name>` の形で書きます。`Name` は通常の `#` 定義名（UpperCamelCase）です。
- 【要決定】ある型が制約を満たしていることの宣言方法。候補は次の2つです。
  1. **ノミナル**: `Player :satisfies Comparable` のような明示宣言を要求する。曖昧性排除の原則には合うが新しいキーワードが必要。
  2. **構造的**: `Self` の各メンバーに合致する実装（`compare: Comparable<Player> a, b -> ...` 等）が見えていれば自動的に満たしたとみなす。新しいキーワードは不要だが、§2.2「1定義＝1実装」との整合（同じ `Player` に対する `Comparable` 実装が複数見える場合の扱い）を詰める必要がある。

## 4. 関数

### 4.1 定義と適用

```
# Greet name: String -> String
greet: Greet name -> 'hello, {name}'

greet 'clum'        // 適用。引数はカンマ区切り
```

### 4.2 部分適用

- arity 未満の適用は残りの引数を待つ関数を返す。
- `(i32, i32 -> i32)` に `i32` を1つ適用すると `(i32 -> i32)`。
- 引数は先頭から順に埋まる。名前指定・飛ばしの適用は持たない（1機能1記法）。

### 4.3 無名関数

```
xs |> map x -> x :mul 2
```

- `引数列 -> 本体`。ブロック本体はインデントで書ける。

### 4.4 パイプ

- `x |> f` は `f x` と同値。`x |> f a` は `f a, x`（最後の引数に差し込む）。末尾差し込みで確定（決定済み）。
- 根拠: 部分適用（§4.2）が「先頭から順に埋まる」仕様であるため、`f a` は「残りの引数を待つ関数」であり、そこへ `x` が最後の引数として入る。パイプのために新しい規則を作るのではなく、部分適用の規則そのものから `|>` の意味が導かれる。
- 複数行のパイプ（決定済み・2026-07-05）: 束縛の右辺などの式に続けて、その式の開始行より1レベル深いインデントで `|>` から始まる行を並べられる。レコード構築の `|>` 継続（§2.7）と同じ形。

```
items = pages
  |> map page ->
    ...
```

## 5. 実行モデル

コアの中で最も重要な区画。**値の評価**と**効果の実行**を分離する。

### 5.1 純粋な式

- 純粋な式（`1 :add 2`, `map f, xs` など）は参照透過であり、評価タイミングは処理系の自由（観測不能なため仕様で縛らない）。

### 5.2 効果と `Eff<T>`

- 効果（I/O、DOM 操作、乱数、時刻取得など）を持つ計算は `Eff<T>` 型の**値**である。作った時点では何も起きない。

```
@js/console
  log: (msg: String -> Eff<Void>)

greet = log 'hello'    // Eff<Void>。まだ何も出力されない
greet |> !             // ここで初めて出力される
```

- `!` は `Eff<T> -> T`。効果の実行は構文上 `!` の位置にのみ存在する。
- `Eff` は値なので、渡す・部分適用に混ぜる・`Vec<Eff<T>>` に並べる、がすべて可能。
- 合成: `Eff<A>` と `(A -> Eff<B>)` をつなぐ演算が必要。【要決定: `|>` を流用するか、専用演算子（`~` の候補）にするか】

### 5.3 実行の順序

- 同一ブロック内の `!` は記述順に実行される。
- `!` を含む式の並べ替え・除去を処理系は行わない。

## 6. スコープ

### 6.1 可視性の原則

値が使えるのは次の2箇所のみ。

1. 自分が生まれたスコープ
2. 明示的に渡された先（引数・子要素としての注入）

外側スコープの暗黙参照（クロージャによる勝手な捕獲）は**コンパイルエラー**。使いたい値は引数で受ける。依存注入は部分適用で行う。

（決定済み・2026-07-04）「自分が生まれたスコープ」の線引きは次のとおり。ファイルのトップレベルにあるもの（`#` 定義とその実装、トップレベルの束縛、import で持ち込んだ名前、prelude）は、そのファイル内のどこからでも——ラムダの内側からでも——参照できる。兄弟ファイルの名前は `@./ファイル名` の明示 import で持ち込む（§8.1。2026-07-18 第3次改訂。同日第2次の「兄弟の暗黙直接参照」は出どころが読めなくなるため同日中に廃止）。禁止されるのは、関数・ラムダの本体で生まれたローカル束縛および引数を、さらに内側のラムダから暗黙に参照することのみ。

### 6.2 `:share`

原則6.1の緩和を明示的に宣言する装置。

```
:share .here   // このファイル内のみ（既定）
:share .tree   // このディレクトリ以下
:share .all    // どこからでも
```

【要決定: `:share` が「定義の可視範囲」なのか「6.1 の捕獲禁止の緩和」なのか、意味論を一本化する】

### 6.3 シャドーイング

- 同名の束縛によるシャドーイングは全面禁止（HTML 要素名の予約と同じ理由）。

## 7. 繰り返しの不在

- ループ構文（for / while）は存在しない。**一般再帰も許可しない**。【要決定】
- 反復はすべて `Vec<T>` / `Stream<T>` に対するコア演算で表現する。

```
map      (A -> B), Vec<A> -> Vec<B>
filter   (A -> Bool), Vec<A> -> Vec<A>
fold     (B, A -> B), B, Vec<A> -> B
each     (A -> Eff<Void>), Vec<A> -> Eff<Void>
```

- 一般再帰を禁止する場合、コア言語は全域性（停止性）に近い性質を持つ。無限の反復が必要な領域（サーバ、ストリーム基盤）は `Stream<T>`（潜在的に無限の列）への演算として表現する。
- この決定は Kafka 的基盤ターゲットの成立可否に直結するため、コア確定前に `Stream<T>` の最小演算セットを検証する。

## 8. モジュール

### 8.1 import（2026-07-18 第2次改訂: `@.` を廃止・別名を導入）

```
@./components
  Card
  MyButton = Button

@js/console
  log: (msg: String -> Eff<Void>)
```

- `@パス` の下に、持ち込む名前をインデントで列挙する。列挙されない名前はスコープへ入らない。
- パスの字句規則（2026-07-18 改訂）: **パスは `@` に密着させる**（`@./components`）。パスは `@` の直後から、次の空白または行末までの連続する非空白文字列。空白を挟む旧形式（2026-07-05）は廃止し、空白があればコンパイルエラー。
- **import の対象は、同じフォルダの兄弟ファイルと、他のディレクトリの窓口**（2026-07-18 第3次改訂）。
  - `@./名前`（自分のフォルダ内） — `名前.clum` という**兄弟ファイル**があればそのファイルの `^` 付き定義を、`名前/` という**サブディレクトリ**があればその窓口 `_.clum`（§8.4）の公開名を持ち込む。両方が存在する場合は曖昧としてコンパイルエラー
  - `@..` で始まるパス（フォルダの外） — ディレクトリ（窓口）だけに解決される。**ファイルの住所は同じフォルダの中でだけ通用する**。どの名前がどのファイルにあるかはそのディレクトリの私事であり、外から見えるのは窓口の公開名だけ（境界の守りは可視性検査。§8.4）。`..` による上りは自由で、**入る側のディレクトリ**の窓口だけが関所になる。複数段の下り（`@./a/b`）の扱いは【要決定31】
  - 旧 `@.`（2026-07-12〜2026-07-18 午前）と兄弟の暗黙直接参照（2026-07-18 第2次）は廃止。名前の出どころは、使う側のファイルの import 列に必ず現れる
- **別名**（2026-07-18 決定。要決定28の解決）: 列挙行は `名前`（そのままの綴りで持ち込む）または `新名 = 元名`（このファイル内だけの綴りに変えて持ち込む）。別名は束縛の記法の再利用であり、新しい記号は導入しない。定義名の別名は UpperCamelCase、値の別名は lower-kebab-case に限る（元と同じ命名カテゴリ）。
- import で列挙する綴りは **export 名そのもの**。`#` 宣言と実装の対は定義名 `Card` で運ばれ、import すると両方が可視になる（**双方向随伴**。決定済み・2026-07-12。要決定19・30の解決）。随伴する実装の可視綴りは、**常にこちら側での定義名を lower-kebab-case にした派生**（`Card` → `card`、`MyButton = Button` と別名にしたら `my-button`。導出規則は §2.4 と同型。2026-07-18 決定）。実装束縛自身の名前（`card: Card ...` の `card`）はファイル私有の綴りであり、境界を越えない。宣言のみ（`# Post`）は `Post`、値のみは束縛名（`index` 等）。1つの export に2通りの綴りは作らない。
- `@js` は JS interop 専用の名前空間。境界では型注釈必須。

### 8.2 export: 定義への `^` 前置（2026-07-18 第2次改訂。`^名前` の独立行形式を同日中に廃止）

`^` は**定義に密着前置する印**で、「この定義をファイルの外（＝同じディレクトリの兄弟）へ差し出す」ことを宣言する。ファイルの中身が名前を語り、`^` がその可視性を1文字で示す。

```
^# Card title: String, body: Html -> Html
card: Card title, body -> ...

^theme: String = 'dark'

helper = ...
```

- `^名前: 型 = 式` — 公開する値束縛。境界のため型注釈が必須（§3.1）。注釈なしはコンパイルエラー。
- `^# Name ...` — 公開する `#` 宣言。実装は双方向随伴で付いてくる（§8.1）。`^` は定義名側にのみ付け、実装束縛に付けるとコンパイルエラー（`^# Name` へ誘導する）。宣言のみ（`# Post`）も `^#` で公開できる。
- `^` なしのトップレベル束縛・宣言は**ファイル私有**。どこからも見えない。
- 公開名はファイル単位の名前空間である。兄弟の2ファイルが同じ名前を `^` してもよい（import 側がファイルの住所で選ぶため曖昧にならない。2026-07-18 第3次改訂）。窓口の差し出し名の一意性は §8.4。
- 旧形式（`^名前` の独立行・2026-07-18 午前、`^` 単独行＋インデント列挙・2026-07-12、`:pub`・それ以前）はすべて廃止。
- ディレクトリの**外**への公開は §8.4 の窓口 `^@` ブロックが担う。`^` 前置はフォルダの中まで、`^@` はディレクトリの外へ、という2段。

### 8.3 Rust によるネイティブ実装

`#` 宣言の実装を、clum 自身の構文ではなく Rust で与えるための窓口です。`@js`（§8.1）と同じ「境界に型注釈を要求する」形を踏襲しますが、`@js` がターゲット（js ターゲット向けの出力）を指すのに対し、`@rust` は**実装の供給元**を指すという点で役割が異なります。ターゲットを問わず（html/css/js/wasm いずれの文脈でも）使えます。

```
@rust ./kernel/add.rs
  add: Self, Self -> Self
```

- clum は「創作活動を楽しくするための言語」という位置づけであり（言語デザイン文書の「はじめに」参照）、パフォーマンスやアルゴリズムの実装そのものを clum の中で完結させることを目的としていません。重い処理・既存の Rust 資産の再利用・処理系の intrinsic（§2.6, §3.3 で触れたプリミティブ数値型の四則演算・比較の実装等）は Rust に委ね、clum は組み合わせと表現に専念する、という分担です。
- **layer 0（native）という位置づけ**: これまでの5層（core / prelude / target / lib / app）は、いずれも clum の構文で書かれる層でした。`@rust` はその外側、Rust で書かれる層への出入り口であり、layer 1（core）自身の intrinsic 実装も、この窓口を処理系が内部的に使っている、という説明になります。一般開発者（layer 4: lib）が同じ窓口を使うことも妨げません。
- 【要決定】`Eff<T>` を含むシグネチャ（副作用を持つ実装）を Rust 側に渡せるようにするか。渡せる場合、Rust 側の副作用と `!` の実行順序保証（§5.3）をどう両立させるかが論点になります。まずは純粋な（`Eff` を含まない）シグネチャのみを対象とする案を軸に検討するのが安全と考えます。
- 【要決定】型の受け渡し（マーシャリング）規則。`i32` 等の組み込み型は自然に対応しますが、`Vec<T>` や `String`、ユーザー定義の直和型・積型をどう Rust 側の型に対応させるかは、`@js` の境界と同様、別途詰める必要があります。

### 8.4 窓口 `_.clum` とエントリファイル（決定済み・2026-07-12。2026-07-18 に1種へ統一・改名、2026-07-20 第4次改訂でエントリを自由名化）

ファイル名が記号のファイルは、コアが役割を定める**役割ファイル**である。識別子名のモジュールと決して衝突せず、単語のファイル名予約（旧 `main.clum` の `main` のような、正当な識別子名の侵食）を作らない。役割ファイルは **`_.clum`（ディレクトリの窓口）の1種のみ**である（2026-07-18。当初の2種体制〔`-.clum` ＝エントリポイント・`^.clum` ＝公開面〕は下記の経緯で統一した）。

**窓口の概念**: 通常のモジュール（`card.clum`）は自分の名前で自分の中身を語るファイルだが、窓口は自分ではなく**ディレクトリの公開面を代弁する**ファイルである。ディレクトリを見る側が知りたいのは「このディレクトリが何を差し出しているか」の1点で、窓口はその列挙だけを持つ。固有の名を持たないため、名の位置には無名の印 `_`（§1.5）を置く。（2026-07-20 第4次改訂でエントリポイントの役はエントリファイルへ移り、予約名が要るのは公開面の1役だけになった。）

役割ファイル名の要件（2026-07-18 改訂で3つに拡張）:

1. 正当な識別子名と衝突しないこと（単語のファイル名予約を作らない。`mod.clum` のような単語名は clum 初の予約語になるため不可）
2. 対話シェルで素打ちしてもシェルエラーにならないこと（2026-07-12 追記）
3. 裸の引数としてコマンドに渡っても安全であること（オプション解釈・グロブ・シェル展開の対象にならない。2026-07-18 追記）

経緯: 当初エントリポイントは `!.clum`（`!`＝実行地点）としたが、`!` は対話シェル（zsh/bash）の履歴展開により素打ちでエラーになる（`touch ./!.clum` → `event not found`。要件2違反）ため、2026-07-12 中に `-.clum` へ改めた。しかし `-.clum` には、`*.clum` のようなグロブ展開が裸の `-.clum` を生成し、受け取ったコマンドがオプションとして解釈するという事故経路が CI に実在する（要件3違反。自分でパスを書く場面の `./-.clum` では防げない）ことが判明した。`^.clum` も zsh の extendedglob 有効時と Windows cmd で要件2が条件付きになる。コアの記号はいずれも要件2か3で落ちる（言語が表現力のある記号として選ぶ文字は、シェルもまた記号として選んでいるため）。外の制約はここまで＝除外条件であり、採用の理由は内側から与える: 窓口はファイルの中身が実体を決める「名を持たないファイル」であり、その名の位置に置くのは無名の印 `_` である（2026-07-18）。

- `_.clum` — ディレクトリの窓口（公開面専用。2026-07-20 第4次改訂）
  - 中身は **`^@` ブロックだけ**。束縛・`#` 宣言・トップレベルの式・取り込みだけの `@` ブロックは書けない（コンパイルエラー）。旧「公開面＋プログラムの同居」（2026-07-18）は第4次改訂で解消し、プログラムはエントリファイルへ移った
  - **公開面は `^@パス` ブロック**（2026-07-18 第2次改訂・同日第3次改訂で供給元の明示を必須化）。`^` ＝出る・`@` ＝入るの合成で「〜から取り込み、そのまま親へ差し出す」と読む。パスの下に差し出す名前を**明示列挙**する。列挙されない名前は出ない
    - `^@./ファイル名` — その兄弟ファイルの `^` 付き定義から差し出す（窓口自身の `^` 付き束縛は差し出せない。窓口はディレクトリを代弁するファイルであり、自前の定義を持つならそれは兄弟ファイルに置く）
    - `^@./サブディレクトリ` — そのサブディレクトリの窓口の公開名から差し出す（**関所の連鎖**の明示形）
    - パスはこの2形のみ（`@./名前` と同じ解決。ファイルとサブディレクトリが同名なら曖昧エラー）。出どころなしの一括形（`^@.`・2026-07-18 第2次）は同日中に廃止——**すべての公開に供給元のパスが付く**
    - 列挙行は `名前` または `新名 = 元名`（別名。§8.1 と同じ規則。定義名の別名の実装は kebab 派生で随伴）
    - 同じ名前を2度差し出す・同じ元を2つの名前で差し出す、はどちらもコンパイルエラー
  - ディレクトリの外から見えるのは窓口の `^@` ブロックに列挙された名前だけ。列挙されない名前（内部ヘルパー等）はディレクトリの外から見えない（package-private）。境界の守りはパスの形ではなく、この可視性検査が担う
  - `^@` ブロックが窓口以外のファイルにあるとコンパイルエラー（親へ差し出せるのは窓口だけ）
  - 窓口を持たないディレクトリは外から import できない
  - `_.clum` はエントリに指定できず、import でも指せない（公開面専用）

```
// prose/_.clum
^@./para
  Para
^@./section
  Section
```

**エントリファイル**（2026-07-20 第4次改訂。予約名を持たない）:

- `clum build <エントリファイル>` が、指定されたファイルのトップレベルの式列を記述順に評価する。効果の実行はソース上の `!` の位置で明示される（§5）
- エントリは**自由な名前の通常ファイル**である。clum はデータ記述言語であり、エントリは「ツールに渡すレシピ文書」にすぎない。呼び出し側がパスで指すため、予約名は不要（`!.clum` → `-.clum` → `_.clum` と続いた予約名の検討は、公開面の1役にだけ残る）
- **トップレベルに式を持つファイル＝プログラムファイル**。式列が評価されるのはエントリとして指定されたときだけで、プログラムファイルは import できない（コンパイルエラー。「唯一のプログラム」の保証は、この検査が担う。要決定24の読み替え）
- 式を持たないファイルをエントリに指定してもよい（束縛の評価のみで、何も書き出さず終了する）
- `Entrypoint` のような型は導入しない。「export された値を処理系が実行する」形にすると、実行地点の `!` がソースから消え、§5 の原則と衝突するため

CLI とビルド:

- `clum build <エントリファイル>`。ディレクトリ指定の旧形式（`clum build <ディレクトリ>`・2026-07-18）は廃止（経緯: 2026-07-12 にファイル指定〔予約名 `main.clum`〕を廃止してディレクトリ指定へ、2026-07-20 に自由名のファイル指定へ改めた。予約名への回帰ではない）。
- CLI はエントリファイル以外の入力（フラグ・OS 環境変数）を持たない（**処理系の規約**であり言語仕様の外縁）。ビルドごとに変わる値（出力先・CDN のベース URL 等）は、環境ごとのエントリファイル（`dev.clum`、`prd.clum` 等）に通常の束縛として書き、環境を知らない共通モジュールへ部分適用で注入する（**運用例**であり規範ではない）。ビルドはソースツリーのみから決まる（暗黙の外部入力を持たない。これは言語側の規範）。

シェル上の注記（規範ではなく既知の性質）:

- `_` はシェル（bash / zsh / fish / PowerShell / cmd）の展開・エスケープの対象にならず、コマンドのオプション解析にも解釈されない。素打ち・裸の引数・グロブ展開のいずれでも不活性である

## 9. ターゲットとコアの境界（2026-07-04 改訂）

- **`:out` は廃止した**（2026-07-04）。ターゲット固有の語彙はブロック宣言で有効化するのではなく、prelude（暫定 prelude を含む）が供給する語彙として**常時可視**とする。ターゲットの分離が必要になった段階では、ファイルの複合拡張子（言語デザイン文書 §3.1: `page.html.clum` 等）で宣言する方針。
- ターゲットが追加できるのは**語彙**（要素名、属性、組み込み型、組み込み関数）のみ。構文規則の追加・変更はできない。この制約が、継ぎ足しても壊れないための規約（言語デザイン文書 §2）の実体であり、`:out` の廃止後も維持される。

12. **`@rust` の副作用の扱いと型マーシャリング規則**（§8.3）。
13. **`'` を含む文字列の表記** — エスケープ手段を持たない現状のままで良いか（§1.4）。
14. **インラインのラムダ本体と適用のカンマ**（フェーズ3実装で発見・2026-07-05） — `map page -> f page, xs` のように引数位置のラムダがインライン本体を持つとき、後続のカンマを (a) 本体側の適用が取り込む（`map` は単一引数のラムダを受け取る）か、(b) 外側の適用の引数区切りとする（`map` は2引数を受け取る）か（§4.1・§4.3）。ステップ1のソースには現れない（ラムダは常に `|>` 経由・ブロック本体）。パーサは暫定で (a) の挙動をテストで固定している。
15. **無名関数の複数引数**（フェーズ3実装で発見・2026-07-05） — §4.3 は `引数列 -> 本体` と定めるが、引数位置に `a, b -> c` を書くとカンマが「外側の適用の引数区切り」と「ラムダの引数列」のどちらか曖昧になる。ステップ1のパーサは無名関数を1引数に限定して曖昧性そのものを回避している。複数引数の無名関数の可否と、許す場合の引数位置での解釈規則が未確定。
16. **レコード構築のフィールド値に許す複数行の式の範囲**（フェーズ3実装で発見・2026-07-05） — HTML 仕様 §1.1 は `element:` の値を「任意の `Html` 式」とするが、フィールド値の位置に複数行の式（子ブロック付き `h`・ネストしたレコード構築・`|>` 継続行）をどこまで書けるかは未定義（§2.7）。ステップ1のパーサは子ブロック付き `h` とレコード構築は受理し、フィールド値での `|>` 継続行は受理しない。
17. **prelude 語彙 `h` と同名の束縛**（フェーズ4実装で発見・2026-07-05） — `h` は特別形式であり値ではないため（HTML 仕様 §3.1）、`h = 1` のような束縛が §6.3 のシャドーイング全面禁止に当たるかが未定義。ステップ1の名前解決は許している（`h .div` は特別形式、密着の `h.div` はフィールドアクセスと、字句上は常に区別できるため衝突はしない）。選択肢: (a) ターゲットの特別形式名を束縛禁止の予約名とする（曖昧性排除の精神に沿う） / (b) 字句で区別可能なため許す（予約名という新カテゴリを増やさない）。
18. **import パスの解決規則**（フェーズ4実装で発見・2026-07-05） — `@ ./index` が「import 元ファイルのディレクトリ基準で `index.clum` を指す」ことは step1 §1 の例からの導出であり、§8.1 に明文がない。拡張子つきパスの可否、ディレクトリ＋index 規約（§2.3 の要決定6）との関係も未定義。ステップ1は「基準ディレクトリ + パス + `.clum` 付与」で実装している。（一部解決・2026-07-12）import の対象はディレクトリのみとなり、ディレクトリを指すパスは窓口 `_.clum` の公開面に照合される。`..` の上りは自由（§8.1）。残る未決は、拡張子つきパスの可否と、複数段の下り `@ ./a/b` の扱い（要決定31）。
19. **定義名（UpperCamelCase）の import 可否**（フェーズ4実装で発見・2026-07-05） — §8.1 の例は `Card`（定義名）を import しているが、`:pub` が付くのは束縛（値・実装）であり、`#` 宣言そのものの export/import の意味論が未定義。コンポーネントを別ファイルから使う場合、タグ解決（HTML 仕様 §2.2）に必要な `#` 宣言をどう持ち込むかが決まっていない。ステップ1の題材（ページ値の import）には現れないため、名前解決は import 名を値の名前空間にのみ登録しており、定義名の import は「export ではありません」エラーになる。選択肢: (a) 実装（または値）の import が対応する `#` 宣言を自動で随伴する / (b) `#` 宣言も `:pub` の対象にできるようにする（1ファイル=1export との関係を要整理） / (c) 定義名 import という概念を持たず、§8.1 の例を改訂する。 → **解決（2026-07-12）: 双方向随伴で解決**（§8.1）。`#` 宣言と実装の対は定義名（`Card`）1つで export / import され、両方が可視になる。1定義＝1実装により対応は一意。
20. **注釈なしトップレベル束縛の前方参照**（フェーズ5実装で発見・2026-07-05） — §6.1 によりトップレベルの名前はファイル内のどこからでも参照でき、名前解決は前方参照を許す。しかしステップ1の型検査（単相・前進推論。HM なし）は、型注釈のない値束縛の型をソース順にしか決められないため、「注釈なし値の使用は定義より後」を要求する非対称がある（注釈つき値・実装・import・prelude は型が先に確定するため前方参照可）。step1 §1 の例（`pages` → `items` → `index`）は定義順なので影響しない。選択肢: (a) 「注釈なしトップレベル束縛の使用は定義の後」を仕様として明文化する / (b) HM 本実装（後続ステップ）の際に依存順の整列または型変数の導入で前方参照を許す。ステップ1は (a) 相当で実装している。
21. **数値の文字列化規則**（フェーズ6実装で発見・2026-07-05） — 文字列補間 `{式}` に数値を埋め込む際の文字列化規則が未定義（§1.4 は数値リテラルの型を定めるのみで、表示形式には触れていない）。特に `f64` の場合、`3.0` を文字列化した結果が `3`（小数点以下を落とす）になるか `3.0`（保持する）になるかが決まっていない。選択肢: (a) Rust の `Display`（`f64::to_string` 等）をそのまま採用する。実装は最も簡単だが、`3.0` → `"3"` のように「整数値に見える小数」の情報が失われる / (b) 型ごとに固定の書式（例: 小数は常に小数点以下を最低1桁残す）を独自に定義する。i32/i64 との対称性や末尾ゼロの扱いなど追加の設計判断を要する。ステップ1の評価器は (a) を暫定として実装している。
22. **ビルド時評価におけるトップレベルの前方参照**（フェーズ6実装で発見・2026-07-05） — §6.1 の可視性原則とフェーズ5の型検査（`seed_globals`）は、トップレベルの `#` 定義の実装（Impl 束縛）と型注釈つきの値束縛について、ファイル内のどこからでも（定義より前の位置からも）参照できることを許している（名前解決・型がいずれも事前に確定できるため）。しかし評価（本フェーズ）はトップレベルの項目をファイルの記述順に逐次評価し、各束縛の値は「その束縛を処理した時点」で初めて評価時の環境に現れるため、型検査は通るはずの前方参照であっても、評価時点でまだ処理されていない束縛を参照すると実行時エラーになり得る（コンパイルは通るのに評価で壊れる、という食い違いが生じ得る）。選択肢: (a) 現状どおり評価は記述順の単純な逐次処理のままとし、この非対称を仕様として明文化して残す（component の実装を使用箇所より前に置くという実務上の書き方であれば問題は顕在化しない） / (b) トップレベルの Impl 束縛・注釈つき値束縛を評価前に2パスで環境へ先に登録し、本体の評価そのものは実際に参照された時点まで遅延させる（サンク相当の遅延評価が必要になり、実装が複雑化する）。ステップ1の評価器は (a) を暫定として実装しており、既存の step1 の例・golden ケースはいずれも定義してから使う順序のため影響は出ない。
23. **整数リテラルの i32 範囲外の扱い**（フェーズ6レビューで発見・2026-07-05） — §1.4 は整数リテラルの既定型を i32 と定めるが、i32 に収まらないリテラル（`2147483648` 等）の扱いが未定義。i64 リテラルの表記手段（サフィックス等）も存在しない。字句解析は i64 として読み型検査が i32 を与えるため、検査なしでは評価時に黙って折り返す（`{2147483648}` が `-2147483648` と出力される）。選択肢: (a) i32 の範囲外はコンパイルエラーとする（i64 値の表記手段は必要になった時点で別途決定する） / (b) 期待型が i64 の位置に限り i64 として解釈する（型駆動の解釈が必要で、HM 本実装との整合を要検討）。ステップ1の型検査は (a) を暫定として実装している。

### 決定済み（このドラフトで確定）

- **インデントは2スペース固定**（§1.1）。
- **数値型は最初から `i32` `i64` `f32` `f64` の4種**。WASM の値型との対応を維持する（§3.2）。
- **`char` 型・文字リテラルは初期セットに含めない**。文字列は `String` のみ。Unicode の単位の決定は必要になった時点で edition 単位で行う（§1.4, §3.2）。
- **パイプ `|>` は末尾差し込み**。部分適用の「先頭から順に埋まる」規則から意味が導かれる（§4.4）。
- **`=>` は廃止し、マッチアームの帰結も `->` に統一**。アームは「束縛を受けて値を返すもの」でありラムダと同じ形のため、専用記号を持たない（§1.5, §3.3）。
- 記号はコアの専有物。ユーザー定義演算子は存在しない（§1.6）。
- **`:` は宣言の記法ではなく呼び出しの記法（§2.4）。** `#` 宣言そのものは常に通常の UpperCamelCase の定義。シグネチャの形（述語／制約／中置記法）が合えば、その定義名を lower-kebab-case にした綴りで `:` 付きでも呼べる。宣言側に新しい構文（`# :name` のような前置コロン）は増やさない。
- **四則演算（`+` `-` `*` `/`）はコアの記号ではない。** `Add` `Sub` `Mul` `Div` という単一メンバーの制約として prelude が通常の `#` 宣言で提供し、`:add` `:mul` 等の呼び出しで使う（§2.6）。副産物として §1.3 の kebab-case と減算の曖昧性が消滅した。
- 述語パターン・型クラス相当の制約は、いずれも §2.4 の応用として統一的に説明される（§3.3, §3.4）。
- **`@rust`（§8.3）を、Rust によるネイティブ実装の窓口として新設。** clum は組み合わせと表現に専念し、重い実装は Rust に委ねるという分担。layer 0（native）として、既存の5層構造の外側に位置づける。
- **`.` の二用法は空白で区別**（§1.7、2026-07-04）。密着した `識別子.識別子` はフィールドアクセス、空白のあとの `.識別子` はドット前置リテラル。
- **暗黙捕獲禁止の範囲を確定**（§6.1、2026-07-04）。トップレベル定義・import・prelude は常時参照可。関数内ローカル束縛と引数のみが禁止の対象。
- **レコード構築は束縛の右辺にも書ける**（§2.7、2026-07-04）。`x = Recipe` の行に続けて、その行より1レベル深いインデントでフィールド列を書く。
- **`:out` を廃止**（§9、2026-07-04）。ターゲット語彙は prelude 供給で常時可視。将来のターゲット分離はファイルの複合拡張子で行う。
- **Vec リテラルはリスト形式（`-`）に一本化**（§2.8、2026-07-04）。カンマ区切り形式は廃止し、型駆動の文脈依存解釈を削除。リスト要素のレコードは型名を書かない（期待型から一意）。
- **`:pub` は独立行**として直後の定義に付く（§8.2、2026-07-04）。（2026-07-12 に `:pub` ごと廃止。公開は `^` ブロックへ）
- **束縛の型注釈 `x: T = 式`** を明文化（§2.2、2026-07-04）。境界では必須（2026-07-12 以降、境界＝`^` に列挙された定義）。
- **タブは全面禁止**（§1.1、2026-07-05）。テキスト行・文字列リテラルの中身のみ例外。
- **文字列の `{` `}` リテラルは `{{` `}}` の対称形**、文字列は1行内（§1.4、2026-07-05）。
- **import パスの字句規則**（§8.1、2026-07-05）。`@` の後の非空白の連続を1つのパスとして読む。
- **複数行のパイプ継続**（§4.4、2026-07-05）。式の開始行より1レベル深い `|>` 行は継続。
- **ファイルは密閉が既定・「1ファイル＝1export」と `:pub` を廃止**（§2.3・§8.2、2026-07-12）。トップレベルの束縛・`#` 宣言はそのままでは外に見えず、公開は `^` ブロックの列挙のみ。コア予約のコロン語は4語になった（§1.6）。
- **`^` は `@` の対**（§1.5・§8.2、2026-07-12）。記号＋名前列挙という同じ形で、`@` が入り・`^` が出。改名 export は存在しない。
- **役割ファイルは記号名の2種**（§8.4、2026-07-12）。`-.clum` ＝エントリポイント（`main.clum` を置換・廃止。`Entrypoint` 型は導入しない）、`^.clum` ＝ディレクトリの公開面（`^` ブロック1つだけのファイル）。記号＝コアの専有物（§1.6）をファイル名へ延長し、単語のファイル名予約を作らない。エントリの当初案 `!.clum` は、`!` が対話シェルの履歴展開で素打ちエラーになるため同日中に `-.clum` へ改めた（役割ファイル名は素打ちでシェルエラーにならないことを要件とする）。（2026-07-18 に窓口 `_.clum` の1種へ統一。下記）
- **import の対象はディレクトリのみ**（§8.1、2026-07-12）。`@ .` で兄弟の公開名を明示 import、`@ パス` でそのディレクトリの窓口の公開面（旧 `^.clum`。2026-07-18 以降は `_.clum`）に照合。ファイルは import で指せない。境界の守りは可視性検査が担う（関所の連鎖。§8.4）。
- **双方向随伴**（§8.1、2026-07-12。要決定19・30の解決）。`#` 宣言と実装の対は定義名1つで運ばれ、宣言のみ（モデル）も定義名で公開できる。
- **CLI は `clum build <ディレクトリ>`・引数なし**（§8.4、2026-07-12。処理系の規約）。環境差分はディレクトリで分け、値はソースに書く（運用例）。ビルドはソースツリーのみから決まる。
- **非エントリモジュールのトップレベルの裸の式を禁止**（§8.4、2026-07-18。要決定24の解決）。トップレベルに式列を持てるのは窓口 `_.clum` のみ。純粋な式も含めて禁止し、効果の実行地点がビルド対象の窓口にのみ存在することを検査で保証する。
- **export は `^名前` の密着行形式**（§8.2・§8.4、2026-07-18）。`^` 単独行＋インデント列挙のブロック形式（2026-07-12）を廃止し、1行1export に統一。窓口の公開面（旧 `^.clum`）も `^名前` 行の列になる。
- **import パスは `@` に密着**（§8.1、2026-07-18）。`@` とパスのあいだの空白を許す形（2026-07-05）を廃止し、`@./components`・`@.` と書く。空白があればコンパイルエラー。行頭記号は引数に密着、という `^名前` と同じ字句の形。
- **役割ファイルを窓口 `_.clum` の1種に統一**（§1.5・§8.4、2026-07-18）。`-.clum`（エントリポイント）と `^.clum`（公開面）の2ファイル体制を廃止し、1つの窓口ファイルが公開面（`^名前` 行）とプログラム（トップレベルの式列）の両方を担う。窓口はディレクトリを代弁するファイルであり、実体はファイルの中身が決めるため固有の名を持たない——名の位置には無名の印 `_` を置く（`_` はコアの記号表に追加。役割ファイル名専用で、言語の字句には現れない）。命名要件に「裸の引数としてコマンドに不活性」を追加した（`-` はグロブ展開が生む裸の `-.clum` をコマンドがオプションと誤解する事故経路が CI に実在するため落ちた。`^` は不採用）。import された窓口の式列は実行されず、式列が動くのはそのディレクトリを直接 build したときのみ。
- **可視性の単位をファイルからディレクトリへ引き直す第2次改訂**（§2.3・§6.1・§8.1・§8.2・§8.4、2026-07-18 第2次）。同日午前の「`^名前` 行・`@.`・窓口の `^名前` 列挙」を同日中に置換した。(1) `^` は定義への密着前置となり、`^` 付き定義は兄弟ファイルから import なしで直接参照できる（`^` なしはファイル私有）。「ファイルは編集上の単位にすぎない」（§2.3）が可視性の面でも字義どおりになった。(2) `@.` は廃止。import は他ディレクトリ専用になった。(3) 窓口の公開面は `^@パス` ブロック（`^@.` ＝自フォルダ、`^@./サブディレクトリ` ＝関所の連鎖の明示形）。供給元の明示により要決定32が構造的に解消。(4) 別名 `新名 = 元名` を `@`・`^@` の列挙行に導入（要決定28の解決）。同じ名前を2度・同じ元を2名で、はエラー。
- **兄弟参照はファイル住所の明示 import に確定（第3次改訂）**（§2.3・§6.1・§8.1・§8.2・§8.4、2026-07-18 第3次）。第2次の「兄弟の暗黙直接参照」を同日中に廃止した。名前の出どころが使う側の import 列に現れないためである。`@./名前` は同じフォルダでは兄弟ファイル `名前.clum`（その `^` 付き定義）を、フォルダの外ではディレクトリの窓口を指す（両方あれば曖昧エラー）。ファイルの住所は同じフォルダの中でだけ通用し、`..` で出るパスはディレクトリのみに解決される（私事の原則はフォルダ境界で効く）。窓口の公開面も出どころ必須の `^@./供給元` に統一し、`^@.` を廃止。兄弟2ファイルの同名 `^` は許可（import 側がファイルで選ぶ）。
- **エントリポイントを自由名のエントリファイルに変更（第4次改訂）**（§8.4、2026-07-20）。`clum build <エントリファイル>` が指定ファイルの式列を実行する。エントリは予約名を持たない通常ファイルで、`dev.clum`・`prd.clum` のように環境ごとのエントリを同じディレクトリに並べられる（環境ディレクトリの運用例を置換）。トップレベルに式を持つファイル＝プログラムファイルは import 不可（「唯一のプログラム」の検査）。窓口 `_.clum` は公開面（`^@` ブロック）専用になり、ディレクトリ指定の CLI（2026-07-18）と「窓口のプログラム役・import された窓口の式列は実行しない」規則は廃止。予約ファイル名が要るのは公開面の1役だけになった。
