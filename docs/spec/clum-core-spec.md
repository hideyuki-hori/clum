# clum コア仕様ドラフト v1

コア層＝全ターゲット（html / css / js / 将来の追加ターゲット）が共有する部分。
ここに含まれるのは **字句・構文・型システム・スコープ・実行モデル・モジュール** のみ。
継ぎ足しても壊れないための規約（言語デザイン文書 §2）により、コアは凍結に近い扱いとする。ここでの決定がすべての土台になる。

**レイヤー構造上の位置づけ**: このドキュメントは 5 層構造（layer 1: core 〜 layer 5: app）のうち layer 1（core）と layer 2（prelude）の両方を扱う。`:pub` `:is` `:any` `:else` `:share` の5語（§1.6。`:out` は 2026-07-04 に廃止）はモジュール・型・スコープの意味論に直結するため layer 1 側に置くが、§2.4 の `# :name` 機構を使って定義される具体的な述語・制約（`:lt`, `:comparable` 等）は layer 2（prelude）に属する。層の全体像は言語デザイン文書 §2 を参照。

【要決定】は提案であり未確定。

## 1. 字句

### 1.1 ソース

- UTF-8。
- インデントはスペースのみ。1レベル＝2スペース固定（決定済み）。
- タブはソースのどこにあってもコンパイルエラー（決定済み・2026-07-05）。例外は html のテキスト行の中身と文字列リテラルの中身のみ（生データとして保持される）。
- コメントは `//` のみ。ブロックコメントは持たない。

### 1.2 識別子

- 定義: UpperCamelCase（例: `Card`, `SaveItem`, `Add`）
- 変数・関数: lower-kebab-case（例: `current-target`, `save-item`）

`:` 呼び出し（§2.4）は独立した命名カテゴリではありません。`:add` は定義名 `Add` を lower-kebab-case にした綴りであり、新しい識別子ではなく既存の定義への別の参照方法です。

### 1.3 kebab-case

`-` は識別子の一部としてのみ現れます。演算子としての `-`（減算）はコアに存在しないため（§1.5, §2.6）、`a-b` は空白の有無にかかわらず常に識別子であり、曖昧性は生じません。

### 1.4 リテラル

- 整数: `42`, `-1`（既定型 `i32`）。負号はリテラルの字句の一部であり、演算子ではない（§1.3, §2.6）。
- 小数: `3.14`（既定型 `f64`）
- 文字列: `'...'` のみ。`"` の文字列構文は持たない（1機能1記法）。
- 補間: 文字列内 `{式}`。リテラルの `{` `}` は `{{` `}}` と書く（単独の `}` もエラー。決定済み・2026-07-05。html のテキスト行の規則と同一の対称形）。
- 文字列リテラルは1行内に収める（改行を含む文字列の表記は持たない。決定済み・2026-07-05）。`'` 自体を文字列に含める手段は【要決定】（§10）。
- 文字リテラルは存在しない（決定済み）。`char` 型自体を初期セットに含めず、文字列は `String` のみで扱う。文字単位の処理は Unicode の単位（コードポイント／書記素）の決定を伴うため、必要になった時点で edition 単位で導入を検討する。

### 1.5 記号の割り当て

- `#` — 定義の宣言
- `@` — import
- `?` — パターンマッチ
- `!` — 効果の実行
- `|>` — パイプ
- `->` — 関数・マッチアームの帰結（決定済み: `=>` は存在しない）
- `<-` — 【要決定: ブロック脱出ガード or Eff 束縛】
- `~` — 【要決定: Eff 束縛 or 未割り当て予約】

**`+` `-` `*` `/` はコアの記号に含まれません。** 四則演算は §2.6 の修飾子（`:add` `:sub` `:mul` `:div`）として prelude（layer 2）が提供します。これにより §1.3 の kebab-case とのあいだの曖昧性そのものが発生しなくなります。

### 1.6 記号と `:xx` の二層構造

拡張性の入り口を1箇所に固定する。

- **記号はコアの専有物。** 上表の記号のみが存在し、ユーザー・ライブラリによる演算子定義は一切できない。新記号の追加はコア変更であり、edition 境界でのみ検討する。
- **`:` 呼び出しは拡張可能。** 通常の `#` 定義（layer 2 が供給する）は、シグネチャの形が合えば `:` 付きの綴りでも呼べます（§2.4）。新しい識別子ではなく、既存の定義名を lower-kebab-case にした綴りにすぎません。
- コア予約5語（`:any` `:is` `:else` `:pub` `:share`）のうち、実際に layer 1 に残るものが何かは §2.4 の各論・以降の検討事項です。layer 1 に残るものは、対応する `#` 定義を持たない、注釈専用の綴りとして扱われます。

「記号＝凍結、コロン語＝ライブラリで成長」という分担が、継ぎ足しても壊れないための規約（言語デザイン文書 §2）の字句レベルでの実体となる。

### 1.7 ドットの二つの用法（決定済み・2026-07-04）

`.` の役割は、前後の空白の有無だけで字句レベルで一意に決まる。

- **フィールドアクセス**: `識別子.識別子`（`.` の前後に空白を挟まない）。`page.title` はレコード `page` のフィールド `title` の読み出し。連鎖できる（`e.current-target.value`）。
- **ドット前置リテラル**: 空白のあとに置く `.識別子`。`h .div` の `.div`（`Tag`）、`:share .all` の `.all` のように、閉じた集合の1要素を指す綴り。

## 2. 定義と束縛

### 2.1 `#` 宣言

`#` は名前に**型（シグネチャ）を与える**宣言。

```
# User name: String, age: i32          // 積型（レコード）
# SaveItem api: Api, item: Item -> Void  // 関数シグネチャ
# Session :any LoggedIn User, Guest      // 直和型
```

### 2.2 実装と束縛

```
save-item: SaveItem api, item ->
  ...

x = 1 :add 2       // 束縛。再代入は存在しない
```

- すべての束縛は不変。`mut` に相当する概念はコアに持たない。
- 束縛には型注釈を付けられる: `x: T = 式`（2026-07-04 決定）。注釈が必須なのは境界（`:pub` 定義等、§3.1）のみ。
- **1定義＝1実装**: 同一スコープから同じ定義の実装が2つ見える状態はコンパイルエラー。コンポーネントタグの定義名解決（HTML 仕様 §2.2）はこの規則に依存する。

### 2.3 1ファイル＝1export

- `:pub` を付けられる定義はファイルにつき1つ。ファイルは責務の境界。
- 複数の公開が必要な場合はディレクトリ＋`index` ファイルで統合する。【要決定: index の規約】

### 2.4 `:` 呼び出し（コアが与える唯一の拡張点）

`:` は**宣言の記法ではなく、呼び出しの記法**でございます。`#` 宣言そのものは何も変わりません。通常の `# Name` / `name: Name ...`（UpperCamelCase の型名と lower-kebab-case の実装名という既存パターン）のままです。

```
# Is a: T, b: T -> Bool
is: Is a, b -> ...
```

このとき、`Is` はシグネチャの**形**に応じて、通常の呼び出し（`Is a, b`）に加えて `:` 付きの呼び出しでも参照できます。`:` に続く綴りは**定義名を lower-kebab-case にしたもの**です（`Is` → `:is`）。

- `..., Self -> Bool` の形（対象が末尾引数）→ `?` のアーム位置で `:is 引数...` として使えます（**述語**、§3.3）。
- `Self` を含む複数メンバーの束 → 型パラメータの境界位置で `<T: is>` として使えます（**制約**、§3.4）。
- メンバーが1つだけの制約 → 中置記法 `a :is b` としても使えます（§2.5）。
- どの形にも当てはまらない定義（`# SaveItem` 等）は `:` 付きでは呼べません。通常の呼び出しのみです。

```
? session
  is Guest -> ...      // 通常呼び出し。コンストラクタ名そのものでも束縛できる（§3.3）
```

- `Self` は `#` 宣言の本体内でのみ使える組み込みの型変数で、「呼び出し側で解決される型」を指します。新しい記号は導入せず、コアが予約する識別子として扱います。
- **命名以外に新しい構文は一切ございません。** `#` 宣言の書き方は1種類のままで、`:` はその定義を呼ぶときの、もう一つの綴りにすぎません。以前の案（宣言名に `:` を前置する、`Modifier` という型を新設する）はいずれも撤回し、この呼び出し規則1つに統合いたします。
- コア予約の5語（`:any` `:is` `:else` `:pub` `:share` のうち、実際に降ろせるものは §1.6 参照）と同名の定義は、同一スコープで両方が見える場合にコンパイルエラーです。
- 【要決定】この規則は「形が合えば自動的に `:` 呼び出しが有効になる」という**暗黙の**振る舞いです。`# IsAdult age: i32 -> Bool` のような、パターンマッチでの使用を意図していない普通の真偽値関数まで自動的に `:is-adult` として呼べてしまう点は、「暗黙の何かを認めない」というコアの精神（§6.1 の暗黙捕獲禁止と同種の懸念）と緊張関係にあります。何らかの明示的な opt-in（例: シグネチャ内の注釈）を要求するかどうかは未確定です。

この一段階の一般化により、§1.6 で述べた「記号＝凍結、コロン語＝ライブラリで成長」という分担を、新規構文を一切増やさずに実現いたします。ライブラリが行うのはすべて既存の `#` 宣言の範囲内の操作でございます。

### 2.5 用途3: 中置記法

§2.4 で述べた「述語」「制約」に加え、制約のうち**メンバーがちょうど1つ**のものは、追加のキーワードなしに中置記法でも使えます。

```
# Add
  add: Self, Self -> Self

x = 1 :add 2
```

- `a :name b` は、`a` の型に対する `Name` の実装が持つ唯一のメンバーの呼び出しへと脱糖されます。
- メンバーが2つ以上ある制約（例: `Comparable` が将来 `compare` 以外のメンバーを持つ場合）は中置記法の対象外で、通常どおり定義名参照（`Comparable.compare a, b` のように、モジュール等の別の場面で使う参照構文を流用）で呼び出します。
- 用途の判定基準がここでも**シグネチャの形のみ**である点は §2.4 と変わりません。中置記法専用の宣言キーワードは導入いたしません。

### 2.6 四則演算はコアの記号ではなく prelude の定義

`+` `-` `*` `/` はコアの記号表（§1.5）に存在いたしません。四則演算は §2.4/§2.5 の仕組みだけを使い、prelude（layer 2）が単一メンバーの制約として、通常の `#` 宣言で提供いたします。

```
# Add
  add: Self, Self -> Self

# Sub
  sub: Self, Self -> Self

# Mul
  mul: Self, Self -> Self

# Div
  div: Self, Self -> Self
```

```
x = 1 :add 2
y = x :mul 3
```

- `i32` `i64` `f32` `f64` は prelude によってこれら4制約の実装を持つ状態で出荷されます。実装そのものは処理系組み込み（intrinsic）であり、`# Add` はその**インターフェース**を宣言しているにすぎません（§3.3 の `Lt` と同様の位置づけです）。
- **既存の演算子方式に対する利点は拡張性です。** `+` `-` `*` `/` が記号として固定されていた場合、独自の数値型（`Vector`, `Money` 等）に対する演算子オーバーロードには専用の言語機構が別途必要になります。この方式であれば、任意の型が同じ `# Add` を実装するだけで `v1 :add v2` が成立し、コアに何も追加いたしません。
- 【要決定】除算は 0 除算を伴い得るため、`div: Self, Self -> Self` のまま（未定義動作／panic 相当）にするか、`div: Self, Self -> Eff<Self>` として失敗を効果として明示するかは未確定です。後者を選ぶ場合、`x :div y` は `Eff<Self>` を返すため `!` を要求する形になり、§5（実行モデル）との整合性が高くなります。
- 比較（`<` `>` `==` 等）についても同様の扱いとするかは別途要決定です。すでに §3.3 で `Lt` を述語の例として使用しておりますので、四則演算と対称的に「比較は述語シグネチャの `#` 定義、算術は単一メンバー制約の `#` 定義」という整理が自然に成り立ちます。

### 2.7 レコード構築（決定済み）

積型（レコード）の値は、**型名（UpperCamelCase）＋インデントしたフィールド列**で構築します。宣言（§2.1）と同じ見た目を構築にも使う、という対応です。

```
# Recipe documents: Vec<Document>

Recipe
  documents:
    - path: './dist/index.html'
      element: index
    - path: './dist/about.html'
      element: about
  |> build
  |> !
```

（例は 2026-07-04 の第2次改訂後の形。`documents:` の値はリスト形式の Vec リテラル、§2.8。）

- フィールドは1行1つ、`名前: 値` の形。カンマは書きません（行がフィールドの区切りです）。
- 全フィールド必須。省略・既定値の仕組みは持ちません（曖昧性排除）。順序は宣言と同一であることを要求します。【要決定: 順序自由を許すか】
- **`|>` による継続**: 構築ブロック内で `|>` から始まる行は、フィールド列の終わりを示し、**構築されたレコード全体**を右辺へ流します。フィールド行は必ず `識別子:` で始まるため、`|>` 行との区別は字句だけで一意に決まります。上の例は `build recipe |> !` と同じ意味であり、`Recipe` の構築 → `build` へパイプ → `!` で実行、と読み下せます。
- インラインの構築記法（`Recipe { documents: ... }` のような1行形式）は**持ちません**。1機能1記法です。
- 構築は束縛の右辺にも書けます。`x = Recipe` のように行の途中から構築を始めた場合、フィールド列は**その行より1レベル深いインデント**で続きます（HTML 仕様の `index = h .html` と同じ規則。2026-07-04 決定の帰結）。

### 2.8 Vec リテラル（2026-07-04 改訂: リスト形式に一本化）

`Vec<T>` の値は、`-` で始まる行を縦に並べた**リスト形式**で書きます（LiveScript / YAML のリストに倣う）。

```
xs =
  - 1
  - 2
  - 3

documents:
  - path: './dist/index.html'
    element: index
```

- 各要素は `- `（ハイフン＋空白。2文字＝インデント1レベル分）で始まります。
- 要素が**レコード**の場合、型名は書かず、最初のフィールドを `- ` と同じ行に、残りのフィールドを最初のフィールドと同じ桁（`-` から1レベル深い位置）に揃えて書きます。要素型はリストの期待型から一意に定まるため、型名の省略に曖昧性はありません。単独の式の位置でのレコード構築は従来どおり §2.7 の型名つきの形であり、場面が重ならないため1機能1記法は保たれます。
- 要素が**1行に収まる式**（リテラル・識別子参照等）の場合は `- 式` を1行で書きます。複数行にわたる式（`h` 式等）を要素に直接書けるかは【要決定】。必要な場合は先に `=` で束縛し、識別子で参照します。
- リストを書ける位置は、**レコードのフィールド値**（`名前:` の直後で改行し、1レベル深く置く）と **`=` 束縛の右辺**（`=` の直後で改行し、1レベル深く置く）です。
- **旧仕様のカンマ区切り形式（`files: index, about` / `xs = 1, 2, 3`）は廃止**（2026-07-04）。「期待型が `Vec<T>` の位置でのみカンマ列を Vec と解釈する」という文脈依存の解釈規則そのものが不要になり、カンマの意味は関数適用の引数区切りだけになります。
- 【要決定: 空の `Vec` の表記】は引き続き未決。

## 3. 型システム

### 3.1 基本方針

- 静的型付け、Hindley–Milner 型推論。
- 型注釈が必須なのは**境界のみ**: `:pub` 定義、JS interop、ターゲット間参照。内部は推論に任せる。

### 3.2 組み込み型

```
i32  i64  f32  f64  String  Bool  Void
Vec<T>
Eff<T>        // 未実行の効果（§5）
```

- 数値型は最初から `i32` `i64` `f32` `f64` の4種（決定済み）。WASM の値型との1対1対応を維持するため（言語デザイン文書 §10）。
- `char` は初期セットに含めない（決定済み、§1.4）。

### 3.3 直和型とマッチ

```
# Session :any LoggedIn User, Guest

? session
  :is LoggedIn user -> ...   // ペイロードの束縛
  :is Guest        -> ...
  :else            -> ...
```

- マッチは網羅性検査あり。全ケースを尽くしている場合 `:else` は書けない（到達不能はエラー）。
- `?` は式であり値を返す。全アームの型は一致しなければならない。

#### 述語パターン（`:` 呼び出しの応用、§2.4）

`:is`（コンストラクタマッチ。実体は §3.3 冒頭のとおりコンストラクタ名の裸の参照）に加え、アーム位置の `:name 引数...` は**述語**として解釈されます。述語は §2.4 のとおり、シグネチャが `..., Self -> Bool` の形をした通常の `#` 定義です。

```
# Lt threshold: i32, Self -> Bool
lt: Lt threshold, target -> ...
  // i32/i64/f32/f64 に対しては処理系組み込みの比較で実装される（intrinsic）

? n
  :lt 0 -> 'negative'
  :else -> 'positive'
```

- `Lt` のようにプリミティブ数値型を対象とする述語は、宣言（インターフェース）は通常の `#` で書かれますが、`i32` 等に対する実際の比較は言語処理系が直接提供する組み込み実装（intrinsic）です。ユーザー定義の述語がさらに別の述語を呼び出して合成することは可能ですが、どこかで必ず組み込み実装に行き着きます（無限後退を避けるための境界です）。
- コアが持つのは §2.4 の呼び出し規則のみで、`Lt` `Gt` `In` のような個々の語彙はすべてライブラリが通常の `#` と実装で供給いたします。`@` で import すれば使えます。
- コア予約のコロン語（§1.6 の5語のうち layer 1 に残るもの）と同名の定義は、同一スコープで両方が見える場合にコンパイルエラーです。
- 網羅性: 述語アームは網羅性検査に**寄与しません**。述語アームを1つでも含むマッチは、コンストラクタで全ケースを尽くすか `:else` を置くかのどちらかが必須です。

#### ブロック脱出ガード 【要決定】

逐次処理の途中に置く1行マッチ。成立時はその値で**現在のインデントブロックを脱出**し、不成立時は次の行へ続く。

```
n = 1 :add 2
n ? :lt 0 <- 'invalid'
...続きの処理
```

脱糖規則（後続行が暗黙の `:else` アームになる）:

```
? n
  :lt 0 -> 'invalid'
  :else ->
    ...続きの処理
```

- 脱出先は関数ではなく**最も内側のインデントブロック**。ラムダ越しに外へ跳ばないため、early return の古典的な曖昧さを持たない。
- ガードの値と、ブロックの通常の最終値は同じ型でなければならない（脱糖後の既存規則から従う）。
- 未決: 記号を `<-` にするか（`Eff` 束縛との取り合い、§1.5）。

### 3.4 ジェネリクス

- `<T>` による型パラメータ。`# Map<A, B> f: (A -> B), xs: Vec<A> -> Vec<B>`
- 型クラス相当の制約は §2.4 のとおり、`Self` を含む複数メンバーの束を持つ通常の `#` 定義で表現します。新しい構文（`:constraint` のような専用キーワード）は導入しません。

```
# Comparable
  compare: Self, Self -> i32

# Max<T: Comparable> a: T, b: T -> T
max: Max a, b ->
  ? Comparable.compare a, b
    :lt 0 -> b
    :else -> a
```

- 型パラメータの境界は `<T: Name>` の形で書きます。`Name` は通常の `#` 定義名（UpperCamelCase）です。
- 【要決定】ある型が制約を満たしていることの宣言方法。候補は次の2つです。
  1. **ノミナル**: `Player :satisfies Comparable` のような明示宣言を要求する。曖昧性排除の原則には合うが新しいキーワードが必要。
  2. **構造的**: `Self` の各メンバーに合致する実装（`compare: Comparable<Player> a, b -> ...` 等）が見えていれば自動的に満たしたとみなす。新しいキーワードは不要だが、§2.2「1定義＝1実装」との整合（同じ `Player` に対する `Comparable` 実装が複数見える場合の扱い）を詰める必要がある。

## 4. 関数

### 4.1 定義と適用

```
# Greet name: String -> String
greet: Greet name -> 'hello, {name}'

greet 'clum'        // 適用。引数はカンマ区切り
```

### 4.2 部分適用

- arity 未満の適用は残りの引数を待つ関数を返す。
- `(i32, i32 -> i32)` に `i32` を1つ適用すると `(i32 -> i32)`。
- 引数は先頭から順に埋まる。名前指定・飛ばしの適用は持たない（1機能1記法）。

### 4.3 無名関数

```
xs |> map x -> x :mul 2
```

- `引数列 -> 本体`。ブロック本体はインデントで書ける。

### 4.4 パイプ

- `x |> f` は `f x` と同値。`x |> f a` は `f a, x`（最後の引数に差し込む）。末尾差し込みで確定（決定済み）。
- 根拠: 部分適用（§4.2）が「先頭から順に埋まる」仕様であるため、`f a` は「残りの引数を待つ関数」であり、そこへ `x` が最後の引数として入る。パイプのために新しい規則を作るのではなく、部分適用の規則そのものから `|>` の意味が導かれる。
- 複数行のパイプ（決定済み・2026-07-05）: 束縛の右辺などの式に続けて、その式の開始行より1レベル深いインデントで `|>` から始まる行を並べられる。レコード構築の `|>` 継続（§2.7）と同じ形。

```
items = pages
  |> map page ->
    ...
```

## 5. 実行モデル

コアの中で最も重要な区画。**値の評価**と**効果の実行**を分離する。

### 5.1 純粋な式

- 純粋な式（`1 :add 2`, `map f, xs` など）は参照透過であり、評価タイミングは処理系の自由（観測不能なため仕様で縛らない）。

### 5.2 効果と `Eff<T>`

- 効果（I/O、DOM 操作、乱数、時刻取得など）を持つ計算は `Eff<T>` 型の**値**である。作った時点では何も起きない。

```
@js/console
  log: (msg: String -> Eff<Void>)

greet = log 'hello'    // Eff<Void>。まだ何も出力されない
greet |> !             // ここで初めて出力される
```

- `!` は `Eff<T> -> T`。効果の実行は構文上 `!` の位置にのみ存在する。
- `Eff` は値なので、渡す・部分適用に混ぜる・`Vec<Eff<T>>` に並べる、がすべて可能。
- 合成: `Eff<A>` と `(A -> Eff<B>)` をつなぐ演算が必要。【要決定: `|>` を流用するか、専用演算子（`~` の候補）にするか】

### 5.3 実行の順序

- 同一ブロック内の `!` は記述順に実行される。
- `!` を含む式の並べ替え・除去を処理系は行わない。

## 6. スコープ

### 6.1 可視性の原則

値が使えるのは次の2箇所のみ。

1. 自分が生まれたスコープ
2. 明示的に渡された先（引数・子要素としての注入）

外側スコープの暗黙参照（クロージャによる勝手な捕獲）は**コンパイルエラー**。使いたい値は引数で受ける。依存注入は部分適用で行う。

（決定済み・2026-07-04）「自分が生まれたスコープ」の線引きは次のとおり。ファイルのトップレベルにあるもの（`#` 定義とその実装、トップレベルの束縛、import で持ち込んだ名前、prelude）は、そのファイル内のどこからでも——ラムダの内側からでも——参照できる。禁止されるのは、関数・ラムダの本体で生まれたローカル束縛および引数を、さらに内側のラムダから暗黙に参照することのみ。

### 6.2 `:share`

原則6.1の緩和を明示的に宣言する装置。

```
:share .here   // このファイル内のみ（既定）
:share .tree   // このディレクトリ以下
:share .all    // どこからでも
```

【要決定: `:share` が「定義の可視範囲」なのか「6.1 の捕獲禁止の緩和」なのか、意味論を一本化する】

### 6.3 シャドーイング

- 同名の束縛によるシャドーイングは全面禁止（HTML 要素名の予約と同じ理由）。

## 7. 繰り返しの不在

- ループ構文（for / while）は存在しない。**一般再帰も許可しない**。【要決定】
- 反復はすべて `Vec<T>` / `Stream<T>` に対するコア演算で表現する。

```
map      (A -> B), Vec<A> -> Vec<B>
filter   (A -> Bool), Vec<A> -> Vec<A>
fold     (B, A -> B), B, Vec<A> -> B
each     (A -> Eff<Void>), Vec<A> -> Eff<Void>
```

- 一般再帰を禁止する場合、コア言語は全域性（停止性）に近い性質を持つ。無限の反復が必要な領域（サーバ、ストリーム基盤）は `Stream<T>`（潜在的に無限の列）への演算として表現する。
- この決定は Kafka 的基盤ターゲットの成立可否に直結するため、コア確定前に `Stream<T>` の最小演算セットを検証する。

## 8. モジュール

### 8.1 import

```
@ ./components/card
  Card

@js/console
  log: (msg: String -> Eff<Void>)
```

- `@ path` の下に、持ち込む束縛をインデントで列挙する。
- パスの字句規則（決定済み・2026-07-05）: `@` の直後（空白を挟んでもよい）から、次の空白または行末までの連続する非空白文字列を1つのパスとして読む。`@ ./components/card` も `@js/console` もこの規則で読める。
- `@js` は JS interop 専用の名前空間。境界では型注釈必須。

### 8.2 export

- `:pub` の付いた定義がそのファイルの export（1ファイル＝1export）。
- `:pub` は**独立した行**として書き、直後の定義に付く（2026-07-04 決定）。`:pub` の付く定義は境界のため型注釈が必須（§3.1）。

```
:pub
index: Html = ...
```

### 8.3 Rust によるネイティブ実装

`#` 宣言の実装を、clum 自身の構文ではなく Rust で与えるための窓口です。`@js`（§8.1）と同じ「境界に型注釈を要求する」形を踏襲しますが、`@js` がターゲット（js ターゲット向けの出力）を指すのに対し、`@rust` は**実装の供給元**を指すという点で役割が異なります。ターゲットを問わず（html/css/js/wasm いずれの文脈でも）使えます。

```
@rust ./kernel/add.rs
  add: Self, Self -> Self
```

- clum は「創作活動を楽しくするための言語」という位置づけであり（言語デザイン文書の「はじめに」参照）、パフォーマンスやアルゴリズムの実装そのものを clum の中で完結させることを目的としていません。重い処理・既存の Rust 資産の再利用・処理系の intrinsic（§2.6, §3.3 で触れたプリミティブ数値型の四則演算・比較の実装等）は Rust に委ね、clum は組み合わせと表現に専念する、という分担です。
- **layer 0（native）という位置づけ**: これまでの5層（core / prelude / target / lib / app）は、いずれも clum の構文で書かれる層でした。`@rust` はその外側、Rust で書かれる層への出入り口であり、layer 1（core）自身の intrinsic 実装も、この窓口を処理系が内部的に使っている、という説明になります。一般開発者（layer 4: lib）が同じ窓口を使うことも妨げません。
- 【要決定】`Eff<T>` を含むシグネチャ（副作用を持つ実装）を Rust 側に渡せるようにするか。渡せる場合、Rust 側の副作用と `!` の実行順序保証（§5.3）をどう両立させるかが論点になります。まずは純粋な（`Eff` を含まない）シグネチャのみを対象とする案を軸に検討するのが安全と考えます。
- 【要決定】型の受け渡し（マーシャリング）規則。`i32` 等の組み込み型は自然に対応しますが、`Vec<T>` や `String`、ユーザー定義の直和型・積型をどう Rust 側の型に対応させるかは、`@js` の境界と同様、別途詰める必要があります。

## 9. ターゲットとコアの境界（2026-07-04 改訂）

- **`:out` は廃止した**（2026-07-04）。ターゲット固有の語彙はブロック宣言で有効化するのではなく、prelude（暫定 prelude を含む）が供給する語彙として**常時可視**とする。ターゲットの分離が必要になった段階では、ファイルの複合拡張子（言語デザイン文書 §3.1: `page.html.clum` 等）で宣言する方針。
- ターゲットが追加できるのは**語彙**（要素名、属性、組み込み型、組み込み関数）のみ。構文規則の追加・変更はできない。この制約が、継ぎ足しても壊れないための規約（言語デザイン文書 §2）の実体であり、`:out` の廃止後も維持される。

## 10. 要決定リスト

1. **`<-` と `~` の割り当て** — ブロック脱出ガードと `Eff` 束縛のどちらにどちらを与えるか（§1.5, §3.3, §5.2）。
2. **ブロック脱出ガードの採否** — 脱糖規則ごと確定するか（§3.3）。
3. **制約の充足宣言方式** — ノミナルか構造的か（§3.4）。
4. **`:share` の意味論** — 可視範囲か捕獲許可か（§6.2）。
5. **一般再帰の禁止** — 禁止で確定するか（§7）。最重要。ステップ1の名前解決は暫定で「ブロックローカル束縛は逐次登録（自己参照・前方参照不可）、トップレベルは先に全登録（§6.1『どこからでも』の帰結として自己参照・前方参照が名前解決上は通る）」という非対称で実装している。この項目の決定時に整合させる。
6. **1ファイル＝1export と index 規約**（§2.3）。
7. **除算の 0 除算の扱い** — `Self, Self -> Self` のままか `Self, Self -> Eff<Self>` にするか（§2.6）。
8. **比較演算子の統一方針** — `Lt` 等の述語シグネチャに一本化するか（§2.6 末尾）。
9. **`:` 呼び出しの暗黙性** — 形が合えば自動的に有効になる仕様のままで良いか、明示的な opt-in を要求するか（§2.4）。
10. **`:is` `:any` の layer 1 残留範囲** — コンストラクタ名の裸の参照（§3.3）に一本化し `:is` 自体を廃止するか、糖衣として残すか。
11. **`:pub` `:share` `:else` の扱い** — 対応する `#` 定義を持たない、layer 1 の注釈専用の綴りとして固定するか。
12. **`@rust` の副作用の扱いと型マーシャリング規則**（§8.3）。
13. **`'` を含む文字列の表記** — エスケープ手段を持たない現状のままで良いか（§1.4）。
14. **インラインのラムダ本体と適用のカンマ**（フェーズ3実装で発見・2026-07-05） — `map page -> f page, xs` のように引数位置のラムダがインライン本体を持つとき、後続のカンマを (a) 本体側の適用が取り込む（`map` は単一引数のラムダを受け取る）か、(b) 外側の適用の引数区切りとする（`map` は2引数を受け取る）か（§4.1・§4.3）。ステップ1のソースには現れない（ラムダは常に `|>` 経由・ブロック本体）。パーサは暫定で (a) の挙動をテストで固定している。
15. **無名関数の複数引数**（フェーズ3実装で発見・2026-07-05） — §4.3 は `引数列 -> 本体` と定めるが、引数位置に `a, b -> c` を書くとカンマが「外側の適用の引数区切り」と「ラムダの引数列」のどちらか曖昧になる。ステップ1のパーサは無名関数を1引数に限定して曖昧性そのものを回避している。複数引数の無名関数の可否と、許す場合の引数位置での解釈規則が未確定。
16. **レコード構築のフィールド値に許す複数行の式の範囲**（フェーズ3実装で発見・2026-07-05） — HTML 仕様 §1.1 は `element:` の値を「任意の `Html` 式」とするが、フィールド値の位置に複数行の式（子ブロック付き `h`・ネストしたレコード構築・`|>` 継続行）をどこまで書けるかは未定義（§2.7）。ステップ1のパーサは子ブロック付き `h` とレコード構築は受理し、フィールド値での `|>` 継続行は受理しない。
17. **prelude 語彙 `h` と同名の束縛**（フェーズ4実装で発見・2026-07-05） — `h` は特別形式であり値ではないため（HTML 仕様 §3.1）、`h = 1` のような束縛が §6.3 のシャドーイング全面禁止に当たるかが未定義。ステップ1の名前解決は許している（`h .div` は特別形式、密着の `h.div` はフィールドアクセスと、字句上は常に区別できるため衝突はしない）。選択肢: (a) ターゲットの特別形式名を束縛禁止の予約名とする（曖昧性排除の精神に沿う） / (b) 字句で区別可能なため許す（予約名という新カテゴリを増やさない）。
18. **import パスの解決規則**（フェーズ4実装で発見・2026-07-05） — `@ ./index` が「import 元ファイルのディレクトリ基準で `index.clum` を指す」ことは step1 §1 の例からの導出であり、§8.1 に明文がない。拡張子つきパスの可否、ディレクトリ＋index 規約（§2.3 の要決定6）との関係も未定義。ステップ1は「基準ディレクトリ + パス + `.clum` 付与」で実装している。
19. **定義名（UpperCamelCase）の import 可否**（フェーズ4実装で発見・2026-07-05） — §8.1 の例は `Card`（定義名）を import しているが、`:pub` が付くのは束縛（値・実装）であり、`#` 宣言そのものの export/import の意味論が未定義。コンポーネントを別ファイルから使う場合、タグ解決（HTML 仕様 §2.2）に必要な `#` 宣言をどう持ち込むかが決まっていない。ステップ1の題材（ページ値の import）には現れないため、名前解決は import 名を値の名前空間にのみ登録しており、定義名の import は「export ではありません」エラーになる。選択肢: (a) 実装（または値）の import が対応する `#` 宣言を自動で随伴する / (b) `#` 宣言も `:pub` の対象にできるようにする（1ファイル=1export との関係を要整理） / (c) 定義名 import という概念を持たず、§8.1 の例を改訂する。

### 決定済み（このドラフトで確定）

- **インデントは2スペース固定**（§1.1）。
- **数値型は最初から `i32` `i64` `f32` `f64` の4種**。WASM の値型との対応を維持する（§3.2）。
- **`char` 型・文字リテラルは初期セットに含めない**。文字列は `String` のみ。Unicode の単位の決定は必要になった時点で edition 単位で行う（§1.4, §3.2）。
- **パイプ `|>` は末尾差し込み**。部分適用の「先頭から順に埋まる」規則から意味が導かれる（§4.4）。
- **`=>` は廃止し、マッチアームの帰結も `->` に統一**。アームは「束縛を受けて値を返すもの」でありラムダと同じ形のため、専用記号を持たない（§1.5, §3.3）。
- 記号はコアの専有物。ユーザー定義演算子は存在しない（§1.6）。
- **`:` は宣言の記法ではなく呼び出しの記法（§2.4）。** `#` 宣言そのものは常に通常の UpperCamelCase の定義。シグネチャの形（述語／制約／中置記法）が合えば、その定義名を lower-kebab-case にした綴りで `:` 付きでも呼べる。宣言側に新しい構文（`# :name` のような前置コロン）は増やさない。
- **四則演算（`+` `-` `*` `/`）はコアの記号ではない。** `Add` `Sub` `Mul` `Div` という単一メンバーの制約として prelude が通常の `#` 宣言で提供し、`:add` `:mul` 等の呼び出しで使う（§2.6）。副産物として §1.3 の kebab-case と減算の曖昧性が消滅した。
- 述語パターン・型クラス相当の制約は、いずれも §2.4 の応用として統一的に説明される（§3.3, §3.4）。
- **`@rust`（§8.3）を、Rust によるネイティブ実装の窓口として新設。** clum は組み合わせと表現に専念し、重い実装は Rust に委ねるという分担。layer 0（native）として、既存の5層構造の外側に位置づける。
- **`.` の二用法は空白で区別**（§1.7、2026-07-04）。密着した `識別子.識別子` はフィールドアクセス、空白のあとの `.識別子` はドット前置リテラル。
- **暗黙捕獲禁止の範囲を確定**（§6.1、2026-07-04）。トップレベル定義・import・prelude は常時参照可。関数内ローカル束縛と引数のみが禁止の対象。
- **レコード構築は束縛の右辺にも書ける**（§2.7、2026-07-04）。`x = Recipe` の行に続けて、その行より1レベル深いインデントでフィールド列を書く。
- **`:out` を廃止**（§9、2026-07-04）。ターゲット語彙は prelude 供給で常時可視。将来のターゲット分離はファイルの複合拡張子で行う。
- **Vec リテラルはリスト形式（`-`）に一本化**（§2.8、2026-07-04）。カンマ区切り形式は廃止し、型駆動の文脈依存解釈を削除。リスト要素のレコードは型名を書かない（期待型から一意）。
- **`:pub` は独立行**として直後の定義に付く（§8.2、2026-07-04）。
