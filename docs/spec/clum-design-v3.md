# clum 言語デザイン v3

> 最小の要件から出発し、後から継ぎ足しても壊れない言語。
> ビルドすると Web 標準の HTML / CSS / JS が出る。

---

## 0. v3 での決定事項

1. 言語名は **clum**（小文字）に統一する。
2. 最小要件: **ビルドしたら HTML / CSS / JS が Web 標準の仕様で出力される**こと。
3. リアクティブ性は入れるが、それは **JS ターゲットの責務**とする。`js.clum` に書いたものの中でリアクティブ性を担保する。
4. 「最小から出発して、後で継ぎ足してもグチャグチャにならない」ことを言語設計そのものの要件とする（§2）。

---

## 1. 設計原則

1. **曖昧性の排除** — 1つの書き方しか存在しない構文を目指す。
2. **インデントは構造のためだけにある** — ネストを表すコンパイル時情報であり、出力には一切残らない。
3. **実行は `!` で明示** — 副作用の発生地点がコード上で必ず見える。
4. **値は生まれた場所と、渡された先でしか使えない** — 暗黙のスコープ透過を認めない。依存は常に明示的に注入される。
5. **一つの構文、複数のターゲット** — HTML / CSS / JS は同一構文の上でターゲットが分かれるだけで、別言語にしない。
6. **出力は Web 標準** — 独自ランタイムを前提とする出力を作らない。出た HTML / CSS / JS は clum なしで読める・動く。

---

## 2. 継ぎ足しても壊れないための規約

言語仕様を5層に分け、機能追加の入り口を制限する（数字が大きいほど上位。依存は必ず上から下への一方向のみ）。

- 5. app — 一般開発者が書くアプリケーション
  - 作り手: 一般開発者
  - 変更の重さ: 自由
- 4. lib — 各ターゲット向けの再利用可能な部品（UI コンポーネント集、`Comparable` 等のドメイン定義、CSS デザインシステム 等）。一般開発者が作るのはここまで
  - 作り手: 一般開発者
  - 変更の重さ: 自由。`@` で import して使うだけの層
  - 育てるものではなく、要件ごとに作り直すケーススタディとして位置づける（下記コンセプト参照）
- 3. target — どこで動くかの語彙（html / css / js / wasm...）。`h`（要素）、CSS プロパティ語彙、JS interop 語彙、wasm 語彙
  - 作り手: 言語処理系（各ターゲットの実装者）
  - 変更の重さ: ターゲット追加は自由。既存ターゲットの語彙変更は影響範囲を要確認
- 2. prelude — `:pub` `:is` `:any` `:else` `:share` 等の基礎修飾子、`Vec` `String` などの基礎型、`map` `fold` などの基礎関数
  - 作り手: 言語処理系（同梱）
  - 変更の重さ: core に準じて重いが、理論上は core の機能だけで書き直せる
- 1. core — 記号（`#` `@` `?` `!` `|>` `->` 等）、§2.4 の `:` 呼び出し規則そのもの（`#` 宣言の形が合えば `:` 付きでも呼べる、という規則）、型システム・型推論、スコープ規則、`Eff`/`!` の実行モデル、モジュールの意味論
  - 作り手: 言語処理系そのもの
  - 変更の重さ: 凍結に近い。edition 境界でのみ変更
- 0. native — Rust による実装（`@rust`）。clum の構文の外側。処理系自身の intrinsic（数値演算・比較等）や、重い処理の実装
  - 作り手: 言語処理系・上級ライブラリ作者
  - 変更の重さ: Rust 側の変更なので clum の edition とは独立

上位層は下位層の機能だけを使って作られ、下位層は上位層の存在を知らない。layer 0 は clum の構文で書かれる5層の外側にあり、`#` 宣言の実装先として使われる（clum コア仕様 §8.3）。

- **layer 1 と layer 2 の境界は「言語の文法として必須か、言語の機能だけで書けるか」で引く。** `:is` や `:pub` はモジュール・型・スコープの意味論に直結するため layer 1 側に置くが、`Lt` のような具体的な述語や `Comparable` のような具体的な制約は、§2.4 の `:` 呼び出し規則さえあれば layer 2 として同梱できる（機構は layer 1、中身は layer 2）。
- **一般開発者が主に触れるのは layer 4 と layer 5。** layer 1〜3 は言語処理系が提供し、layer 4 から `@` で import して使う想定。
- **layer 4（lib）は使い捨てのケーススタディとして扱う。** 機能を長期的に育てるのではなく、要件が生じた都度、その要件専用のライブラリを（AI に生成させる等の方法で）作り直す運用を想定する。これは本規約 3（1機能1記法・後から良い記法を思いついたら追加ではなく置換）を layer 4 の運用にまで延長した考え方である。
- **layer 0（native）は「予約語をなくす」という目標とは別軸。** clum は創作活動を楽しくするための言語であり、パフォーマンスやアルゴリズムの実装そのものを担うことを目的としない。重い実装・処理系自身の intrinsic は Rust（layer 0）に委ね、clum は組み合わせと表現に専念する。

1. **コアは凍結に近い扱いにする** — 構文・型・スコープ規則への追加は最後の手段。新機能はまず layer 2 以上として設計する。
2. **直交性の強制** — 既存機能の組み合わせで書けるものを構文にしない（糖衣構文を安易に増やさない）。
3. **1機能1記法** — 同じことを書く方法を2つ作らない。後から良い記法を思いついた場合は追加ではなく置換（バージョンで区切る）。
4. **ターゲットは足し算で増やす** — 将来のターゲット追加（例: サーバ、ストリーム基盤）は layer 1（core）に手を入れずに成立しなければならない。html/css/js の3ターゲットはその設計が正しいかの試金石。
5. **廃止の手続きを最初から決めておく** — edition（世代）を仕様に含め、破壊的変更は edition 境界でのみ許す。

## 3. ターゲットモデル

名前の検討（2026-07-04）: この概念を platform / runtime に改める案を検討し、**target のまま維持**と決定した。

- runtime は設計原則6「独自ランタイムを前提とする出力を作らない」と語が衝突し、実行系を持たない静的出力（html / css）にも合わない。
- platform は「どこで動くかの語彙」という層の定義には合うが、css 単体を指すにはやや広義。
- コンパイル出力の種類を指す用語として正確な target を維持する。

### 3.1 ファイル＝ターゲットの単位

ファイル名の複合拡張子でターゲットを宣言する。

```
page.html.clum   → page.html
page.css.clum    → page.css
page.js.clum     → page.js
```

- 1ファイルは1ターゲットにのみ属する。ターゲットの混在ファイルは作らない。
- コア構文（`#` 定義、`|>`、`!`、`?` マッチ、`@` import、部分適用、スコープ規則）は全ターゲット共通。
- ターゲットごとに使える語彙（要素、プロパティ、組み込み型）が変わる。

### 3.2 各ターゲットの責務

- `html.clum`
  - 出力: 静的 HTML
  - 責務: 文書構造
  - 持たないもの: 状態・イベント・動的更新
- `css.clum`
  - 出力: 標準 CSS
  - 責務: スタイル
  - 持たないもの: ロジック
- `js.clum`
  - 出力: 標準 JS (ESM)
  - 責務: 振る舞い・状態・**リアクティブ性**
- `wasm.clum`（計画）
  - 出力: 標準 WebAssembly
  - 責務: 計算負荷の高い純粋ロジック
  - 持たないもの: DOM・I/O（js 側から呼ばれる）

- `html.clum` の出力はビルド時に確定する純粋な HTML。JS が無効でも文書として成立する。
- リアクティブな UI は `js.clum` が DOM を構築・更新することで実現する。要素記法（§5）は html / js 両ターゲットで同じに見えるが、コンパイル結果が「静的 HTML」か「DOM を構築する JS」かが異なる。

### 3.3 ターゲット間の参照

- `js.clum` から `html.clum` が出力した要素への参照（マウントポイント）の取り方は未決（§9）。
- `css.clum` のクラス名を html / js 側から型付きで参照できるようにするかも未決（§9）。

## 4. コア構文（全ターゲット共通）

- 定義: `# Name ...` — 型・関数シグネチャの宣言
- パイプ: `|>` — 左の値を右の関数へ
- 明示実行: `!` — 効果の実行はここでのみ起きる
- 直和型: `:any A, B, C`
- パターンマッチ: `? x` + `:is` / `:else`
- import: `@ path` + インデントで束縛列挙
- 可視性: `:share .all / .tree / .here`, `:pub`
- 識別子（定義）: UpperCamelCase（例: `# SaveItem`）
- 識別子（変数・関数）: lower-kebab-case（例: `current-target`, `save-item`）
- 文字列補間: `'{x}'`
- コメント: `//` のみ
- ループなし: ストリーム演算（`map` 等）で代替

### 4.1 スコープモデル

値（関数を含む）が使えるのは次の2箇所だけ。

1. 自分が生まれたスコープ
2. 明示的に渡された先（引数・子要素として注入された場合）

### 4.2 部分適用

arity 未満の適用は残りの引数を待つ関数を返す。全引数を適用しても実行はされず、実行は常に `!` でのみ起きる。

```
# A x: i32, y: i32, z: i32 -> Void
a: A = ...

x-binded   = a 1
xy-binded  = x-binded 2
xyz-binded = xy-binded 3
```

依存注入は部分適用で行う。

## 5. html ターゲット

### 5.1 要素・テキスト・属性（2026-07-04 改訂）

```
h .div
  h .p
    こんにちは
    このインデントはコンパイルされるときえます

h .a href '/about', class 'nav-link'
  About

h .input type 'text', disabled
```

- 要素は `h .tag` 方式で書く（2026-07-04 決定。詳細は HTML 仕様）。子は `h` 行の直下の1レベル深いインデントブロックで、`->` は書かない。
- 裸の行はテキストノード。インデント・改行は出力に残らない。
- 属性は `名前 値`、複数はカンマ区切り。子を持たない要素は子ブロックを書かない。
- 補間 `{}` は**ビルド時に解決できる値のみ**許す（実行時の値は js ターゲットの領分）。

### 5.2 静的な制御

`?` マッチとストリーム演算はビルド時データに対して使える（静的サイトの列挙など）。

```
items = pages
  |> map page ->
    h .li
      h .a href page.path
        {page.title}

h .ul
  {items}
```

（2026-07-05 決定: 子ブロック内の裸の式行は書けない。式はトップレベルで束縛し `{式}` で埋め込む。）

## 6. css ターゲット

プロパティは属性と同じ `名前 値` 形式。1行1プロパティ。

```
.card ->
  padding 16px
  border-radius 8px
  color '#333'

.card:hover ->
  background '#f5f5f5'
```

- セレクタは要素名の位置に書く。
- 出力は標準 CSS。独自の実行時スタイル機構は持たない。
- 値の型付け（`16px` を長さ型にするか）は未決（§9）。

## 7. js ターゲット

### 7.1 責務

状態・イベント・DOM の構築と更新、すなわち**リアクティブ性のすべて**は js ターゲットに置く。html ターゲットには一切漏らさない。

### 7.2 要素記法とイベント

html ターゲットと同じ見た目で書くが、コンパイル結果は DOM を構築する JS になる。
（注: 以下の例は旧記法のまま。2026-07-04 の決定で要素記法は `h .tag` 方式に統一されたため、js ターゲットの設計時にこの節の例も改訂する。）

```
block ->
  handle e ->
    v = e.current-target.value
    log v |> !
  div ->
    こんにちは
    button on-click handle ->
      ボタン
```

- `on-*` 属性は js ターゲットでのみ使える。
- `e.current-target` は要素の種類に応じて型付けされる（`input` なら `.value: String`）。

### 7.3 リアクティブ性（機構は未決）

リアクティブ性を js.clum の中で担保するという責務配置のみ確定。機構は未決:

- **案A: クロージャのみ** — 状態は生まれたスコープに閉じ、更新関数を明示的に配る。スコープモデルと整合。再描画単位の特定が課題。
- **案B: signal 組み込み** — `Signal<T>` を組み込み型にし購読箇所だけ差分更新。暗黙の購読は原則4と緊張関係にあるため、入れるなら購読も明示にする。

いずれの案でも、出力はフレームワーク非依存の標準 JS（原則6）。

### 7.4 JS interop

```
@js/console
  log: (msg: String -> Void)

@js ./legacy.js
  init-analytics: ( -> Void)
```

- 境界では型注釈必須。
- `:pub` の定義は ESM named export として出力される。

## 8. ビルドモデル

```
src/
  index.html.clum ─┐
  style.css.clum  ─┼─ clum build ─→ dist/index.html, dist/style.css, dist/index.js
  index.js.clum   ─┘
```

- 出力はすべて Web 標準（HTML5 / 標準 CSS / ESM）。バンドラや独自ランタイムを要求しない。
- 効果（イベント登録・DOM 更新）は `!` の位置に対応するコードにのみ現れる。

## 10. 長期方針

- clum は html / css / js の3ターゲットから出発し、ターゲット追加によって汎用言語へ育てる。
- **wasm ターゲットをスコープに含める。** 最小要件（html / css / js）には入れないが、コア設計は wasm 出力を破綻なく追加できることを制約として持つ。
  - コアの数値型（`i32` `i64` `f32` `f64`）は WASM の値型と一致しており、この対応は維持する。
  - 効果を持たない純粋関数のみが wasm 化の対象。`Eff<T>` を含む定義は wasm ターゲットに置けない（ホスト＝js 側の責務）。この分割はコアの純粋／効果の分離（`!` / `Eff`）がそのまま境界線になる。
- ストリーム基盤（Kafka 的なもの）を言語で作る構想は、コアのストリーム演算・`!` によるコミットポイントの明示・明示注入スコープの延長線上にあるターゲットとして扱う。コアに手を入れずに成立しない場合、それはコア設計の欠陥として扱う（§2-4）。
