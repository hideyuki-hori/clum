# clum → HTML 仕様ドラフト v1

clum のページ定義（`Html` 値）が標準 HTML にコンパイルされる際の規則。

## 1. ソース構造（2026-07-04 改訂・同日第2次改訂）

ページファイルは通常の clum ファイルであり、ページを `Html` 型の値として束縛し `:pub` で公開する。**`:out .html` ブロックは廃止した**（2026-07-04 第2次改訂）。

```
:pub
index: Html = h .html
  h .head
    h .meta charset 'utf-8'
    h .title
      clum
  h .body
    h .ul
      h .li
        h .a href '/about'
          About
```

- **要素記法は `h .tag` 方式が正**（決定済み・2026-07-04）。裸の要素名（`div ->`）による旧記法は廃止した。
- ページは `Html` 型の**値**として束縛する（`index: Html = h .html ...`。ゼロ引数関数 `= ->` は用いない）。`:pub` は独立行として直後の定義に付き（コア仕様 §8.2）、境界のため型注釈が必須（コア仕様 §3.1）。
- **旧 `Element` 型は `Html` に改名した**（決定済み・2026-07-04。型は1つのみ）。
- html ターゲットの語彙（`h`・`Tag`・`Html`・`Document`・`Recipe`・`build`・属性）は**暫定 prelude の一部としてどのファイルでも常時可視**（決定済み・2026-07-04）。`map` や `true` と同格であり、import は不要。コア構文（`#`, `|>`, `!`, `?`, `@`, 部分適用、スコープ規則）はそのまま使える。

### 1.1 エントリポイントと出力ファイル: `main.clum`（マニフェスト方式・2026-07-04 第2次改訂）

出力の対象と場所は、命名規約による暗黙の判定ではなく、`main.clum` に**明示的に列挙**します。暫定 prelude が提供する `Recipe` / `Document` 型と `build: Recipe -> Eff<Void>` を使い、レコード構築（コア仕様 §2.7）からパイプで `build` へ流し、`!` で実行します。

```
# Recipe documents: Vec<Document>
# Document path: String, element: Html
```

```
@ ./index
  index
@ ./about
  about

Recipe
  documents:
    - path: './dist/index.html'
      element: index
    - path: './dist/about.html'
      element: about
  |> build
  |> !
```

- `main.clum` は**ビルド時に処理系が実行する唯一のプログラム**です。トップレベルの `!` はビルド時に走ります。ファイル書き出しという効果が `!` の位置に明示される形であり、実行モデル（コア仕様 §5）との整合が取れています。
- `documents:` の値はリスト形式の Vec リテラル（コア仕様 §2.8）です。各 `Document` が出力1ファイルに対応し、`path` に出力先を、`element` に出力する `Html` 値を明示します。
- **`path` は `main.clum` のあるディレクトリを基準**に解決します（実行時のカレントディレクトリに依存しない。決定済み・2026-07-04）。
- `element:` の値は任意の `Html` 式です（識別子参照には限定しません）。
- `path` に出力先を自由に書けるため、旧仕様の「識別子名からの出力名導出」とその前提の「`files:` は識別子参照のみ」の制約、および要決定だった「定義名と異なる出力名・サブディレクトリへの出力の指定方法」は、**第2次改訂ですべて不要になりました**。
- ルート要素が `h .html` の場合、`<!DOCTYPE html>` を自動で先頭に付与します。それ以外は fragment として出力します。
- 旧提案（`:pub` の定義名がそのまま出力ファイル名になる暗黙の規約）は廃止しました。「暗黙の何かを認めない」というコアの精神に、マニフェスト方式のほうが忠実であるためです。

## 2. 行の解釈規則（2026-07-04 改訂）

以下の規則が適用されるのは **`h`（およびコンポーネントタグ）の子ブロック内**である（`:out` の廃止に伴い、この構文自体が html 記法の有効範囲を定める）。子ブロック内の1行は、**行頭トークンの字句クラスのみ**で解釈が決まる。名前解決（スコープ検索）の結果によって行の種類が変わることはない。

- `h` — 要素（§3）
- UpperCamelCase の識別子（＝定義名） — コンポーネント呼び出し（§2.2）
- `{` — 値の埋め込み（§4.3）
- `?` — パターンマッチ（§5.1）
- `|>` — 直前の式の継続（パイプ）
- 上記以外 — テキストノード

### 2.1 Tag の検査（2026-07-04 改訂）

- 要素名は識別子ではなく、`h` に渡すドット前置リテラル（`.div` 等。コア仕様 §1.7）である。`Tag` は WHATWG HTML Living Standard の要素名を列挙した閉じた直和型として暫定 prelude が与える。
- `Tag` に存在しない綴り（`.dvi` 等）は**コンパイルエラー**。編集距離の近い既知の Tag があればサジェストを添える。
- 要素名が識別子でなくなったため、旧仕様の「要素名の予約語化」「テキスト行への typo 警告」は不要になった（決定済み・2026-07-04）。

### 2.2 コンポーネントタグは定義名（UpperCamelCase）で書く

命名規則は「定義＝UpperCamelCase、変数・関数＝lower-kebab-case」であるため、コンポーネントの実装関数（`card` など）は小文字となり、字句だけではテキスト行と区別できない。そこでマークアップ上のコンポーネントタグには**実装関数名ではなく定義名を書く**（決定済み・2026-07-04）。

```
# Card title: String, body: Html -> Html
card: Card title, body -> ...

page: Html = h .body
  Card 'お知らせ'              // 定義 Card を参照 → 実装 card に解決される
    h .p
      本日は晴天です
```

- タグ位置の UpperCamelCase 識別子は「その定義の実装」への参照として解決される。
- 子ブロックの規則は `h` と同じ（§3.1。`->` は書かず、タグ行より1レベル深いインデントブロックが子）。子ブロックは、シグネチャ末尾の `Html`（または `Vec<Html>`）引数として渡される。
- この解決が一意であるためには **1定義＝1実装** の制約が必要（1ファイル＝1export の原則と整合する）。同一定義に複数の実装が可視である場合はコンパイルエラー。
- 実装関数 `card` を値として扱う場面（部分適用、引数渡し）では通常どおり lower-kebab-case の名前を使う。タグ記法は html ターゲットの糖衣ではなく「定義経由の呼び出し」という別の解決規則として仕様化する。

## 3. 要素（2026-07-04 改訂）

### 3.1 基本形

```
h .名前 属性1 値1, 属性2 値2
  子...
```

- **`h` は関数値ではなく、html ターゲットが与える特別形式**（決定済み・2026-07-04）。`Tag`（ドット前置リテラル）、任意個の「属性名 値」ペア、直下のインデントブロック（子リスト）を構文として受け取る。`h` 自体の部分適用・値渡しはできない。
- **子ブロックに `->` は書かない**（決定済み・2026-07-04）。`h` の行より**1レベル深いインデントブロック**が子リストになる。レコード構築（コア仕様 §2.7: 型名＋インデントのフィールド列）と同じ形である。`h` が行の途中から始まる場合（`index = h .html` 等）も、その行より1レベル深いブロックが子。
- 子を持たない要素は子ブロックを書かない: `h .div` → `<div></div>`。

### 3.2 void 要素

WHATWG の void 要素（`area, base, br, col, embed, hr, img, input, link, meta, source, track, wbr`）は:

- 子ブロックを持てない。子を書くとコンパイルエラー。
- 属性なしの場合は `h .br` のように単独で要素になる。
- 出力は `<br>` 形式（`/>` は付けない。HTML 構文で出力し、XHTML 構文は採らない）。

### 3.3 属性

- `名前 値` を空白で区切り、複数属性はカンマ区切り。
- 真偽属性は名前のみ: `h .input type 'text', disabled` → `<input type="text" disabled>`。
- 値に使えるのは**ビルド時に確定する式**のみ: リテラル、スコープ内の値、補間文字列。
- `on-*` 属性は html ターゲットでは**使用不可**（コンパイルエラー）。イベントは js ターゲットの責務。
- 属性の型は要素ごとに定義される（`a` に `href: String`、`img` に `src: String, alt: String` など）。未知の属性はコンパイルエラー。ただし `data-*` と `aria-*` は任意に許可する。
- `img` の `alt` など、標準上必須の属性の欠落は警告とする。【要決定: エラーに格上げするか】

### 3.4 出力時のエスケープ

- 属性値は `"` で囲み、値中の `&`, `"`, `<` は文字参照にエスケープする。
- エスケープを回避する手段（raw 出力）は**提供しない**。最小要件に含めず、必要になった時点で edition 単位で検討する。【要決定】

## 4. テキストノード

### 4.1 基本

行頭トークンがどの字句クラスにも該当しない行はテキストノード。

```
h .div
  こんにちは
  このインデントはコンパイルされるときえます
```

- テキスト中の `<`, `>`, `&` は自動エスケープされる。
- `{` `}` をテキストとして書く場合は `{{` `}}` とする。
- 行頭トークンが `h` や UpperCamelCase 識別子になるテキストは、そのままでは要素・コンポーネントと解釈される。そのようなテキストは `{'h で始まる文'}` のように文字列の埋め込み（§4.3）で書く。

### 4.2 連結規則（決定済み: 無挿入）

同一ブロック内の連続するテキスト行は、**何も挿入せず**連結される（改行・インデントは消える）。

```
h .div
  aa
  bb
  cc
```

出力: `<div>aabbcc</div>`

- 日本語コンテンツでは自然な連結になる。欧文で単語間スペースが必要な場合は、行末に明示的に書く。
- 「インデントと改行は構造のための情報であり、出力には残らない」という原則2の最も素直な帰結である。

### 4.3 値の埋め込み

- テキスト内の `{式}` はビルド時に評価され、文字列化・エスケープされて埋め込まれる。
- 行全体が `{式}` の場合、式の型が `Html` / `Vec<Html>` ならノードとして埋め込まれる。文字列ならテキストノード。

```
h .li
  {user.name}
```

## 5. 静的制御

いずれも**ビルド時データ**に対してのみ使える。

### 5.1 分岐

```
? page.kind
  :is Article -> h .article
    ...
  :else -> h .div
    ...
```

各アームは `Html` を返す。

### 5.2 列挙

```
h .ul
  pages
    |> map page ->
      h .li
        h .a href page.path
          {page.title}
```

- `Vec<Html>` は子の位置に置くと展開される。
- js ターゲットで必要な `key` 属性は、html ターゲットでは不要（静的出力のため）。

## 6. 出力形式

- 文字コードは UTF-8。
- 整形（インデント・改行の付与）は行わない（決定済み: 無整形が既定）。ソースの構造情報は出力に一切残さない（原則2の帰結）。HTML では要素間の空白が描画に影響するため、整形は出力の意味を変えるリスクがあり、無整形を既定とすることでこの問題を構造的に回避する。人間可読な整形が必要な場合は、外部フォーマッタ（prettier 等）の適用か、将来のビルドオプションで対応する（後日判断）。
- `//` コメントは出力に残らない。HTML コメント（`<!-- -->`）を出力する構文は最小要件では提供しない。【要決定】

### 決定済み（2026-07-04 追加）

第1次（要素記法まわり）:

- **要素記法は `h .tag` 方式**。裸の要素名記法は廃止（§1）。
- **子ブロックに `->` は書かない**。`h`（およびコンポーネントタグ）の行より1レベル深いインデントブロックが子（§3.1）。
- **`h` は特別形式**。部分適用・値渡しは不可（§3.1）。
- **フィールドアクセスとドット前置リテラルは空白で区別**（コア仕様 §1.7）。
- **暗黙捕獲禁止の範囲**はコア仕様 §6.1 のとおり確定（トップレベル・import・prelude は参照可）。

第2次（`:out` 廃止・マニフェスト刷新。第1次の一部を置換）:

- **`:out .html` を廃止**。html 語彙は暫定 prelude として全ファイルで常時可視（§1、コア仕様 §9）。
- **`Element` を `Html` に改名**（型は1つのみ。§1）。
- **ページは `:pub`（独立行）＋型注釈つき値束縛** `index: Html = h .html ...`（§1、コア仕様 §8.2）。
- **`Recipe` は `documents: Vec<Document>` のみ**（`root` は持たない）。`# Document path: String, element: Html`（§1.1）。
- **Vec リテラルはリスト形式（`-`）に一本化**（コア仕様 §2.8）。
- **出力先は各 `Document.path` に明示**（`main.clum` のディレクトリ基準）。識別子名からの出力名導出・`files:` の識別子参照制約・「別名出力・サブディレクトリ出力」の要決定は不要になり解消（§1.1）。
