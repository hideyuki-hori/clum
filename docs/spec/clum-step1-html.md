# clum ステップ1: HTML 出力のみ

最初の実装ステップのスコープ定義。
**「clum のソースをビルドすると静的 HTML が出る」**、この1点だけを動かす。
prelude の設計は別途行うため、このステップでは実装に必要な最小限の語彙を**暫定 prelude** として処理系に直書きし、後で正式な prelude に差し替える。

---

## 1. このステップで動かすもの

```
// index.clum（ページ）
:pub
index: Html = h .html
  h .head
    h .meta charset 'utf-8'
    h .title
      clum
  h .body
    h .ul
      pages
        |> map page ->
          h .li
            h .a href page.path
              {page.title}
```

```
// main.clum（マニフェスト）
@ ./index
  index

Recipe
  documents:
    - path: './dist/index.html'
      element: index
  |> build
  |> !
```

（2026-07-04 第2次改訂: `:out .html` は廃止し、ページは `:pub`（独立行）＋型注釈つき値束縛 `index: Html = h .html ...` で書く。旧 `Element` 型は `Html` に改名。出力先は `Document.path` に明示する。）

```
clum build ./main.clum → dist/index.html
```

## 2. core のうち、このステップに含める範囲

- 字句（インデント2スペース、`//`、kebab-case、UpperCamelCase）
  - 理由: すべての土台
- リテラル（`String`・数値4種・`Bool`）、文字列補間 `{}`
  - 理由: テキスト・属性値の表現
- `#` 宣言・束縛・不変性・1定義＝1実装
  - 理由: コンポーネント定義
- 関数・部分適用・無名関数 `->`
  - 理由: `h .div` のエイリアス、`map` へのラムダ
- `|>`（末尾差し込み・決定済み）
  - 理由: 列挙の表現
- `@` import・`:pub`・1ファイル＝1export
  - 理由: ページとコンポーネントの分割
- レコード構築（§2.7）・Vec リテラル（§2.8）
  - 理由: `main.clum` の `Recipe`
- `Eff<T>`・`!`（**最小限**: `build: Recipe -> Eff<Void>` の1箇所のみ）
  - 理由: ファイル書き出しは効果であり、`main.clum` の `|> !` で明示する
- スコープ規則（暗黙捕獲禁止）
  - 理由: 最初から効かせないと後で直せない
- `:pub`（独立行）・型注釈つき束縛 `x: Html = 式`
  - 理由: ページの公開境界（境界では型注釈必須）
- HM 型推論（このサブセットに必要な範囲）
  - 理由: 静的型付けの成立

## 3. core のうち、このステップに含めないもの

- `?` マッチ・`:is`・`:else`・直和型
  - 理由: HTML の静的生成に分岐は不要（決定済み: ステップ1から除外）
- `Eff` の**合成**（`~` / `<-` の割り当て、依存する効果の連鎖）
  - 理由: 効果は `build` の1箇所のみで、合成が発生しない
- `:` 呼び出し（述語・制約・中置記法）
  - 理由: 分岐はコンストラクタマッチだけで書ける。四則演算も HTML 生成には必須でない
- ジェネリクスの制約境界 `<T: Name>`
  - 理由: `:` 呼び出しと同じ理由
- `Stream<T>`・generator
  - 理由: 有限の `Vec<T>` で足りる
- `:share`
  - 理由: 最小構成では既定スコープのみで書く
- `@js`・`@rust`
  - 理由: interop はターゲットが増えてから
- js / css / wasm ターゲット
  - 理由: ステップ2以降

`Eff` と `:` 呼び出しという、要決定リストの重い項目の多く（旧2・4・8〜10・12・13相当）がこのステップに**そもそも登場しない**。紙上で決めきれない項目を先送りしたまま、動くものを作れる構成である。

## 4. 暫定 prelude（処理系直書き、後で差し替え）

- `Vec<T>` と `map`（`filter`・`fold` は必要になるまで入れない）
- `Bool` の値 `true` / `false`
- html ターゲット語彙: `h`、`Tag`（`.div` 等の直和型）、`Html`（旧 `Element`）、標準属性、`Document`、`Recipe`、`build: Recipe -> Eff<Void>`
- 暫定 prelude の語彙は import 不要で全ファイルから常時可視（2026-07-04 決定）

## 5. このステップの完了条件

1. 上記 §1 のソースと `main.clum` がエラーなくコンパイルされ、正しい HTML が出力される。
2. 暗黙捕獲・シャドーイング・未知の要素名（`Tag` に無い `.dvi` 等）が**コンパイルエラーになる**ことを確認できる。
3. HTML 仕様ドラフトの確定済み規則（エスケープ、void 要素、テキスト連結）が実装と一致する。

## 7. Rust 実装方針（決定済み）

- パーサ: 手書きの再帰下降。字句解析器がインデントを INDENT/DEDENT トークンへ変換する
- 型検査: まずは単相＋`map` の intrinsic 特別扱いで開始。HM の本実装は後続ステップ（後で ok）
- 位置情報: AST 全ノードに span を**最初から**持たせる
- CLI: `clum build ./main.clum`。エントリファイルは引数で明示する（暗黙のカレント探索はしない）。`clum watch ./main.clum` 等のサブコマンドも同じ形式で追加していく
- テスト: golden test（入力 `.clum` と期待 `.html` の対）を最初のコミットから置く。HTML 仕様の確定済み規則（エスケープ・void 要素・無挿入連結・無整形出力）をそのままケース化する
